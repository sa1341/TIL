# Function Calling과 API 설계

챗봇에 "서울 날씨 알려 줘"라고 물었더니 AI가 **알아서 날씨 API를 호출**했습니다. 기존에는 클라이언트가 분기를 태웠죠 — `if`문이나 라우터가 어떤 API를 호출할지 결정했습니다. 그런데 **Function Calling**이 되면서 그 **결정권이 LLM에게** 넘어갔습니다.

![Function Calling — LLM이 JSON 스키마를 보고 함수 선택|697](../Attached%20file/ai_function_calling.svg)

## 작동 방식

- 개발자가 **사용 가능한 함수 목록을 JSON 스키마로** 넘기면,
- LLM이 **사용자 의도에 맞는 함수와 파라미터를 골라서** 돌려줍니다.

여기서 중요한 건 **description(설명)**입니다. 이게 그냥 문서가 아니라, **LLM이 이 함수를 쓸지 말지 판단하는 근거**가 됩니다.

- `"도시의 현재 날씨 조회"`라고 쓰면 "서울 날씨"에 반응하고,
- `"기상 데이터 조회"`라고 쓰면 같은 함수인데 다른 질문에 반응합니다.

## 새로운 설계 원칙

**API 스키마 자체가 LLM의 정확도를 좌우**합니다. REST API를 설계할 때 **"사람이 읽기 좋은 이름"**을 고민했다면, 이제는 **"AI가 판단하기 좋은 이름"**을 고민해야 합니다.

여러 툴을 조합하는 패턴도 가능합니다. LLM이 **첫 번째 함수 결과를 보고 다음에 어떤 함수를 호출할지 스스로 결정**합니다 — 이게 **AI 에이전트의 시작점**입니다.

## 툴이 많아지면 생기는 문제

- 툴이 수십 개가 넘어가면 **LLM이 비슷한 함수들을 헷갈려** 합니다. `getUserInfo`와 `getUser`가 둘 다 있으면 AI도 확신을 못 합니다.
- "그럼 툴을 합쳐서 줄이면?" — 여기서 **기존 REST 설계와 충돌**이 생깁니다. **REST는 작고 명확한 엔드포인트를 지향**하는데, **Tool Use는 적을수록** 좋습니다.
- 그래서 **같은 서버에 사람용 REST API와 AI용 Tool API를 따로 두는** 팀이 늘고 있습니다.

## 정리

> Function Calling은 **API 설계의 기준 자체를 바꿉니다.** **사람이 호출하는 API와 AI가 고르는 API는 설계 원칙이 다릅니다.**
