# 코드베이스 RAG — 청킹·검색·에이전틱 탐색

LLM에게 "우리 회사 코드베이스에 대해 물어보고 싶다"는 순간 벽에 부딪힙니다. 파일이 수천~수만 개인데 모델의 컨텍스트 창은 유한하고, 전부 밀어 넣을 수도 없습니다. 그래서 **질문과 관련된 코드 조각만 골라 프롬프트에 넣어 주는** 방식이 필요합니다. 이것이 코드베이스에 적용하는 **RAG(Retrieval-Augmented Generation, 검색 증강 생성)**입니다.

문서 RAG와 원리는 같지만, **코드에는 자연어에 없는 문법 구조**가 있어서 그대로 가져다 쓰면 잘 동작하지 않습니다. 이 노트는 청킹 → 인덱싱 → 검색으로 이어지는 파이프라인과, 2025~2026년 현재의 최신 흐름을 정리합니다.

## 1. 청킹 — 어디서 자르느냐가 전부

RAG의 첫 단계는 코드를 검색 가능한 작은 조각(청크)으로 쪼개는 것입니다. 문제는 **어디서 자르느냐**입니다.

![코드 청킹 전략 — 고정 크기 vs AST 구조 인식|697](../Attached%20file/ai_code_chunking.svg)

- **고정 크기 청킹(naive)**: "100줄마다" 혹은 "500토큰마다" 기계적으로 자릅니다. 구현은 쉽지만, 함수 하나가 청크 경계에서 두 동강 나기 일쑤입니다. 검색으로 반쪽짜리 로직만 걸려 오면 LLM이 문맥을 오해합니다.
- **구조 인식 청킹(AST 기반)**: 코드를 **추상 구문 트리(AST)**로 파싱해 **함수·클래스·메서드 같은 문법 단위**로 자릅니다. 청크 하나가 곧 하나의 완결된 의미 단위가 되므로 검색 정확도가 올라갑니다.

큰 함수는 재귀적으로 더 쪼개고, 너무 작은 조각(import 몇 줄 등)은 이웃과 합쳐(greedy merge) **크기 상한 안에서 문법적 완결성을 유지**하는 게 요령입니다. 2025년 발표된 **cAST** 연구는 이 구조 청킹이 고정 크기 방식보다 검색·생성 양쪽에서 일관되게 낫다는 것을 실험으로 보였습니다.

## 2. 인덱싱과 검색 — 파이프라인 전체 그림

청크를 만들었으면 나중에 빠르게 찾을 수 있도록 **인덱스**를 만들고, 질문이 들어오면 그 인덱스에서 **관련 청크를 회수(retrieve)**합니다.

![코드 RAG 파이프라인 — 인덱싱과 질의|697](../Attached%20file/ai_code_rag_pipeline.svg)

인덱싱(오프라인)과 질의(온라인)를 나눠 보면 각 기법의 자리가 분명해집니다.

**① 인덱싱 단계 — 커밋마다 갱신**
- **맥락 주입(Contextual Retrieval)**: 청크를 임베딩하기 전에, "이 코드가 어떤 모듈·클래스의 일부인지" 한두 문장 요약을 청크 앞에 붙입니다. 조각만 보면 무슨 맥락인지 알 수 없는 문제를 보완하는 Anthropic의 기법입니다.
- **이중 인덱스**: **벡터 인덱스**(임베딩으로 *의미가 비슷한* 코드를 찾음)와 **BM25 인덱스**(변수명·함수명 등 *키워드가 정확히 일치*하는 코드를 찾음)를 함께 만듭니다. 코드는 식별자 이름이 결정적 단서라 키워드 검색이 의외로 강력합니다.
- 임베딩은 코드 특화 모델(예: **voyage-code**, **Gemini** 계열)이 범용 텍스트 모델보다 유리합니다.

**② 질의 단계 — 요청마다**
- **하이브리드 검색**: 벡터 검색과 BM25 결과를 합쳐 후보 K개를 회수합니다. 의미 유사도와 정확 일치의 장점을 모두 취합니다.
- **리랭킹(reranking)**: 회수한 후보를 질문과의 관련도로 다시 정렬해 **정말 필요한 상위 몇 개만** 프롬프트에 넣습니다. 상위 몇 개를 남길지(top-5/10/20)는 정확도와 비용·지연 사이의 트레이드오프입니다.

Anthropic 실험 기준으로 맥락 주입(Contextual Embedding + Contextual BM25)만으로 검색 실패가 **약 49% 감소**, 여기에 리랭킹까지 더하면 **약 67% 감소**했습니다.

> 참고로 코드베이스가 작다면(대략 **20만 토큰 이하**) 굳이 RAG 인프라를 만들지 말고 **통째로 컨텍스트에 넣는 편**이 더 정확하고 단순할 수 있습니다. RAG는 "다 넣을 수 없을 때"의 해법입니다.

## 3. 최신 흐름 — 임베딩 인덱스 vs 에이전틱 검색

가장 큰 변화는 **"과연 사전 인덱싱이 꼭 필요한가"**라는 질문입니다. 코드 문맥을 확보하는 방식이 두 갈래로 갈립니다.

![코드 문맥 확보 — 임베딩 인덱스 vs 에이전틱 검색|697](../Attached%20file/ai_code_retrieval_approaches.svg)

- **임베딩 인덱스 RAG**: 위에서 설명한 방식. 미리 청킹·벡터화해 두므로 검색이 빠르고 대규모에 잘 확장됩니다. 대신 **인덱스가 코드 변경을 따라가지 못하는 staleness**와 **청크 경계에서의 문맥 손실**이 약점입니다. Cursor, Sourcegraph 같은 코드 도구들이 이 방식을 씁니다.
- **에이전틱 검색(agentic search)**: 사전 인덱스 없이, **LLM 에이전트가 그 자리에서 `grep`·`glob`·파일 읽기를 반복**하며 필요한 코드를 직접 찾아 나갑니다. **Claude Code가 대표적**입니다. 항상 최신 코드를 읽으므로 staleness가 없고 인덱스 인프라도 필요 없지만, 매 질의마다 탐색하니 **느리고 토큰 소모가 큽니다.**

**어느 쪽이 이겼다고 말하기 어렵고, 둘 다 2026년 현재의 실전 방식입니다.** 그래서 최근엔 섞어 씁니다 — 임베딩으로 후보 파일을 빠르게 좁힌 뒤, 에이전트가 실제 파일을 열어 확인하는 하이브리드가 늘고 있습니다.

이 밖에 **GraphRAG**도 코드에 잘 맞습니다. AST에서 뽑아낸 **호출 그래프·의존 그래프**를 함께 인덱싱하면, "이 함수를 바꾸면 어디가 영향을 받나" 같은 **구조적 질문**에 강해집니다. 단순 유사도 검색으로는 잡기 어려운 관계입니다.

## 정리

> 코드 RAG의 핵심은 **문법 구조를 존중하는 것**입니다. 자연어처럼 아무 데서나 자르면 안 되고(AST 청킹), 조각만으로는 문맥을 알 수 없으니 맥락을 주입하고(Contextual Retrieval), 의미와 키워드를 함께 검색해(하이브리드) 관련도로 추립니다(리랭킹).

실무 선택 기준을 한 줄로:
- **작은 코드베이스** → 통째로 컨텍스트에 넣기
- **크고 자주 안 바뀜** → 임베딩 인덱스 RAG (빠름)
- **최신성이 중요하고 정확한 파일 탐색이 필요** → 에이전틱 검색
- **둘 다** → 임베딩으로 좁히고 에이전트로 확인하는 하이브리드
