# 프로세스 vs 스레드 vs 코루틴 (1만 개의 차이)

스레드 1만 개를 만들었더니 서버가 **OutOfMemory 에러**로 죽었습니다. 그런데 코틀린 **코루틴**으로 똑같이 1만 개를 만들면 멀쩡합니다. 둘 다 동시에 실행되는 건데 왜 이런 차이가 날까요?

![프로세스 vs 스레드 vs 코루틴 비교](../../Attached%20file/cs_process_thread_coroutine.svg)

## 프로세스 — 격리, 그러나 비싸다

- 프로세스는 **완전히 격리된 실행 단위**입니다.
- 메모리를 공유하지 않으니 안전하지만, 새로 만들 때마다 **메모리 전체를 복사**해야 해서 비용이 큽니다.

## 스레드 — 가볍지만 스택이 있다

- 스레드는 프로세스 안에서 돌며 **힙을 공유**하니 프로세스보다 가볍습니다.
- 하지만 **스레드마다 스택을 1MB씩** 잡습니다. **1만 개면 스택만 10GB**입니다.
- 게다가 OS가 스레드를 번갈아 실행할 때마다 **컨텍스트 스위칭**이 발생합니다.

## 코루틴 — OS 스레드 위에서 돈다

- 코루틴은 **OS 스레드 위에서** 돕니다. 실제 OS 스레드는 몇 개만 쓰고, 그 위에서 **코루틴 수만 개를 번갈아** 실행합니다.
- 전환을 **OS가 아니라 런타임**이 하니, 레지스터 저장·복원 같은 비용이 거의 없습니다.

## 코루틴이 만능은 아니다 — 스케줄링의 차이

- 코루틴이 빛나는 건 **IO 대기** 상황입니다. 네트워크 응답을 기다리는 동안 다른 코루틴으로 전환하니 효율적이죠.
- 하지만 **CPU를 꽉 잡고 계산하는 작업**에서는 전환할 틈이 없습니다.
  - **OS 스레드**는 타임슬라이스가 끝나면 OS가 강제로 CPU를 빼앗습니다 → **선점형 스케줄링**
  - **코루틴**은 스스로 양보해야 다음 코루틴이 돕니다 → **협력적 스케줄링**. CPU 바운드 작업이 양보하지 않으면 나머지가 전부 멈춥니다.

## 정리

> **IO 대기가 많으면 코루틴, CPU 연산이 많으면 OS 스레드, 격리가 필요하면 프로세스** — 무엇을 쓸지는 **작업의 성격**이 결정합니다.
