# JSON을 Protobuf로 바꿨더니 70% 줄었다 (직렬화)

서비스 간 통신에서 JSON을 **Protobuf**로 바꿨더니 **같은 데이터의 크기가 70% 줄었습니다.** 같은 정보를 담고 있는데 왜 이렇게 차이가 날까요?

![JSON vs Protobuf 직렬화 비교](../../Attached%20file/cs_json_vs_protobuf.svg)

## 직렬화란

**직렬화(Serialization)**는 메모리에 있는 객체를 **바이트 배열로 바꾸는 것**입니다. 네트워크로 보내거나 파일에 저장하려면 메모리 구조 그대로는 안 되니까요.

## JSON — 텍스트 기반

JSON은 **텍스트 기반 직렬화**로, 사람이 읽을 수 있는 문자열로 바뀝니다.

- 숫자 `123400`을 정수로 저장하면 4바이트면 되는데, JSON은 **문자열이라 6바이트**를 씁니다.
- 거기다 **필드 이름, 콜론, 따옴표, 중괄호**가 전부 바이트를 차지합니다.

## Protobuf — 바이너리 기반

- **필드 이름 대신 번호**를 쓰고, 숫자를 **바이너리로 직접** 저장합니다.
- 구분자나 따옴표도 없습니다. 그래서 같은 데이터가 훨씬 작아집니다.

**속도 차이도 큽니다.** JSON은 문자열을 파싱해야 합니다(따옴표 찾고, 이스케이프 처리하고, 숫자 문자열을 정수로 변환). Protobuf는 **바이트를 그대로 읽으면 끝**입니다.

## 그럼 전부 Protobuf로? — 그렇게 간단하지 않다

1. **디버깅** — JSON은 로그에 찍으면 바로 읽을 수 있지만, Protobuf는 바이너리라 **별도 도구 없이는 내용을 확인할 수 없습니다.**
2. **스키마 관리** — Protobuf는 **필드 번호로 데이터를 식별**하기 때문에, 한 번 쓴 번호를 삭제 후 재사용하면 **이전 버전 클라이언트가 데이터를 잘못 읽습니다.** JSON은 필드 이름이 데이터에 포함되어 있어 이런 문제가 없습니다.

## 정리

> **JSON이 느린 건 "사람이 읽을 수 있는" 대가**입니다. **성능이 필요한 내부 통신은 Protobuf, 유연성이 필요한 외부 통신은 JSON** — 용도에 따라 나누는 것이 실무의 선택입니다.
