# VPC (Virtual Private Cloud)

VPC는 **사용자의 AWS 계정 전용 가상 네트워크**를 의미하며, AWS 서비스의 네트워크 연결을 제어하는 기능입니다. AWS 계정을 생성하면 기본 VPC가 자동으로 생성되고, EC2 인스턴스를 만들면 이 VPC에 연결됩니다.

VPC를 통해 인스턴스가 속하는 네트워크를 구분하고, 각 네트워크에 맞는 설정(IP 대역, 라우팅, 방화벽 등)을 부여할 수 있습니다. 만약 VPC 없이 인스턴스를 마구 생성한다면, 인스턴스끼리 구분이 없어 시스템 복잡도가 크게 올라갈 것입니다. VPC로 네트워크를 논리적으로 격리하면 이런 문제를 깔끔하게 정리할 수 있습니다.

## VPC 구성요소

VPC는 독립된 하나의 네트워크를 구성하기 위한 가장 큰 단위입니다. 각 **리전에 종속**되며, RFC 1918에서 정의한 사설 IP 대역에 맞춰 설계합니다.

VPC에서 사용하는 사설 IP 대역은 아래와 같습니다.

- `10.0.0.0 ~ 10.255.255.255` (10/8 prefix)
- `172.16.0.0 ~ 172.31.255.255` (172.16/12 prefix)
- `192.168.0.0 ~ 192.168.255.255` (192.168/16 prefix)

VPC에서 한번 설정된 IP 대역은 수정할 수 없으며, 각 VPC는 독립적이라 서로 통신할 수 없습니다. 통신하려면 뒤에 설명할 **VPC 피어링**이 필요합니다.

- **서브넷(Subnet)**: VPC 내부를 더 잘게 나눈 네트워크 단위. EC2 인스턴스는 서브넷에 속하며, 서브넷의 IP 대역에서 프라이빗 IP를 할당받습니다. 인터넷과 통신하는 **퍼블릭 서브넷**과, 외부에서 직접 접근할 수 없는 **프라이빗 서브넷**으로 나눠 설계하는 것이 일반적입니다(예: 웹서버는 퍼블릭, DB는 프라이빗).
- **가용영역(AZ)**: 하나의 리전에는 여러 AZ가 있고, 서브넷은 특정 AZ에 속합니다. 서브넷을 여러 AZ에 분산 배치하면 하나의 AZ에 장애가 나도 서비스를 유지할 수 있습니다.
- **인터넷 게이트웨이(IGW)**: VPC를 인터넷과 연결하는 관문. 퍼블릭 서브넷의 라우팅 테이블이 IGW를 향하도록 설정해야 인터넷 통신이 가능합니다.

## ENI (Elastic Network Interface)

- 가상 네트워크 인터페이스로, IP 주소·MAC 주소 등이 부여됩니다.
- 인스턴스에 연결되어 실제 네트워크 통신을 담당합니다.
- 인스턴스 생성 시 기본 네트워크 인터페이스가 IP 등의 정보와 함께 생성됩니다.
- EC2 하나에 추가로 여러 개의 ENI를 연결할 수 있습니다(최대 개수는 인스턴스 유형에 따라 다름).
- 하나 이상의 보안 그룹을 연결할 수 있으며, **동일한 가용영역**의 인스턴스에만 연결 가능합니다.

## CIDR (Classless Inter-Domain Routing)

CIDR은 IP 주소 범위를 정의하는 방식입니다. IP 주소를 작은 그룹으로 분할하는 것을 **서브네팅**이라고 합니다.

- ex1) `192.168.32.0/24` — `/24`가 서브넷 마스크. 앞 24비트가 고정이고 나머지 8비트가 호스트 → **256개**의 IP 사용 가능 (192.168.32.0 ~ 192.168.32.255)
- ex2) `192.168.0.0/16` — **65,536개** (192.168.0.0 ~ 192.168.255.255)
- ex3) `192.168.32.0/25` — **128개** (192.168.32.0 ~ 192.168.32.127)

> prefix 숫자가 작을수록 IP 대역이 넓고, 클수록 좁아집니다. http://www.subnet-calculator.com/cidr.php 에서 자동 계산이 가능합니다.

## Security Group vs NACL — 방화벽 두 종류

AWS의 네트워크 방화벽은 **적용 범위**와 **동작 방식**이 다른 두 가지가 있습니다. 자주 헷갈리므로 표로 비교합니다.

| 구분 | Security Group (SG) | NACL |
|---|---|---|
| 적용 대상 | **인스턴스 수준** (EC2/ENI) | **서브넷 수준** |
| 상태 추적 | 상태 저장(Stateful) — 요청 허용 시 응답 자동 허용 | 상태 비저장(Stateless) — 인/아웃 규칙을 각각 지정 |
| 규칙 종류 | **허용(allow)만** 가능 | 허용 + **거부(deny)** 모두 가능 |
| 연결 관계 | 하나의 ENI에 여러 SG 연결 가능 | 한 서브넷에 하나의 NACL, 하나의 NACL은 여러 서브넷에 연결 가능 |

> NACL이 상태 비저장이라는 점이 핵심입니다. 인바운드 트래픽에 대한 응답도 별도로 아웃바운드 규칙에서 허용해 줘야 합니다.

## VPC Peering

- VPC 간에는 기본적으로 통신이 되지 않습니다.
- VPC 피어링은 두 VPC 사이에 트래픽을 라우팅할 수 있게 해주는 네트워킹 연결입니다.
- 피어링 트래픽은 퍼블릭 인터넷을 통과하지 않고 **프라이빗 IP**로 통신합니다.
- 같은 리전, 다른 리전, 다른 AWS 계정 간에도 피어링이 가능합니다.

**구성 순서**: ① 요청자가 피어링 요청 생성 → ② 수락자 VPC 소유자가 요청 수락 → ③ 양쪽 라우팅 테이블에 피어링 연결 추가.

**지원되지 않는 구성**

- **CIDR 중첩**: IPv4 CIDR 블록이 겹치는 VPC 간에는 피어링을 만들 수 없습니다.
- **전이적 피어링 불가**: A–B, A–C가 피어링돼 있어도 B와 C는 A를 거쳐 통신할 수 없습니다. 피어링은 **1:1 연결**만 가능합니다.
