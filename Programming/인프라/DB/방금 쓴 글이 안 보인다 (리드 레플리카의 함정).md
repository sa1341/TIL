# 방금 쓴 글이 안 보인다 (리드 레플리카의 함정)

"방금 쓴 글이 보이지 않습니다." 사용자 클레임이 **트래픽 피크 시간마다** 들어오기 시작했습니다.

DB는 보통 부하를 나누려고 **쓰기 전용 마스터**와 **읽기 전용 리드 레플리카(Read Replica)**로 분리해 둡니다. 이 서비스도 코드가 그 구조를 따르고 있었죠. 분명 잘 설계된 구조 같은데, 왜 사용자만 자기 글이 안 보일까요?

## 원인 — 복제 지연

원인은 **복제 지연(replication lag)**이었습니다.

- 마스터의 변경 내용은 리드 레플리카로 **곧바로 가지 않고, 비동기 복제**를 거칩니다.
- 사용자 입장에서는 이렇게 흘러갑니다: **글을 쓴 직후** → 새로고침이 **리드 레플리카로 라우팅** → 그때까지 **복제가 끝나지 않음** → 자기 글이 안 보임.

## 해결의 출발 — "방금 쓴 사용자"를 인지하기

해결의 출발은 **방금 쓴 사용자를 인지하는 것**입니다.

- 쓰기 직후 **짧은 윈도우** 동안은 **마스터에서 읽도록** 라우팅을 강제합니다. (read-your-writes)

그런데 이 패턴에도 한계가 있습니다.

- 사용자 인지 정보가 **인스턴스 간에 공유되지 않으면**, A 서버에서 쓴 사용자가 B 서버로 들어왔을 때 다시 레플리카를 타게 됩니다.
- 또 마스터로 강제되는 트래픽이 늘면, 처음 레플리카를 분리한 의미가 흐려집니다.

## 진짜 해법 — 데이터별 일관성 등급

결국 **데이터마다 일관성 등급을 다르게 두는 것**이 운영의 시작입니다.

- 글 쓰기 직후 **본인 화면** → **강한 일관성**
- **인기글 목록** → 몇 초 지연 허용

**한 가지 정책으로 모든 데이터를 다루지 않는 것**이 핵심입니다.

## 정리

> 리드 레플리카는 **읽기 분산일 뿐, 일관성을 보장하지 않습니다.** 복제 지연을 전제로, 데이터마다 필요한 일관성 등급을 나눠서 다뤄야 합니다.
