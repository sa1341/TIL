# 격리 수준과 처리량의 트레이드오프

DB 격리 수준은 **강할수록 안전하다**고 생각하기 쉽습니다. 정합성이 보장되니까요. 그런데 결제·예매 시스템 운영자들은 오히려 **일부러 격리 수준을 낮췄습니다.** 격리를 약하게 하면 정합성이 깨질 텐데, 왜 일부러 낮출까요?

## 격리 수준은 "다이얼"이다

격리 수준은 **동시에 실행되는 트랜잭션이 서로를 얼마나 볼 수 있게 할지**를 정하는 다이얼입니다.

- 약하게 풀면 **빠르지만** 이상한 데이터를 볼 수 있고,
- 강하게 잠그면 **일관성은 좋지만** 서로 기다립니다.

이 다이얼이 막거나 허용하는 건 **세 가지 이상 현상**입니다 — Dirty Read, Non-Repeatable Read, Phantom Read. SQL 표준은 어떤 이상 현상까지 막을지를 **네 단계**로 정해 놨습니다.

- PostgreSQL·Oracle·SQL Server → **READ COMMITTED**가 기본
- MySQL → **REPEATABLE READ**가 기본 (InnoDB는 여기서 **갭 락**으로 Phantom까지 추가 차단해 표준보다 한 단계 강함)

## 문제 — 강할수록 락이 늘어난다

- **Serializable**은 사실상 직렬 실행이고,
- **REPEATABLE READ**도 InnoDB에선 **갭 락으로 다른 트랜잭션의 INSERT 자체를 막습니다.**
- 동시성이 높아지면 **락 대기와 데드락이 폭증**합니다.

**격리 수준 한 단계 차이가 처리량을 몇 배씩 가릅니다.** 열 명이 쓰는 사내 시스템이면 차이가 안 보이지만, **초당 수천~수만 건 결제 시스템**에선 그 한 단계가 SLA를 결정합니다.

## 그럼 약한 격리만 쓰면 끝일까?

아닙니다. 격리를 낮추면 **DB가 막아주던 이상 현상이 그대로 노출**됩니다. DB가 안 막아주니 **애플리케이션이 메워야** 합니다.

- 격리 수준은 **다이얼**이고, **부족한 만큼은 동시성 패턴(애플리케이션 락 등)으로 보완**하는 게 정석입니다.

## 운영팀의 의사결정

- **트래픽이 적을 때** → 강한 격리로 코드를 단순하게
- **트래픽이 늘면** → 격리를 낮춘 뒤, 부족한 정합성은 락 몇 줄로 메움

## 정리

> **강한 격리는 "안전"이 아니라 "비용"이고, 낮은 격리는 "위험"이 아니라 "책임 이전"입니다.** 격리 수준은 트래픽 규모에 맞춰 조절하는 다이얼이며, 낮춘 만큼의 정합성은 애플리케이션이 책임집니다.
