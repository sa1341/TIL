# 인덱스 구조 및 탐색

## 인덱스란?

인덱스는 대용량 테이블에서 필요한 데이터만 빠르고 효율적으로 액세스하기 위해 사용하는 오브젝트입니다. **책 뒤쪽의 색인(찾아보기)**과 같은 역할을 합니다. 색인이 없으면 책을 처음부터 끝까지 넘겨봐야 하지만(= 테이블 풀 스캔), 색인이 있으면 원하는 단어의 페이지로 바로 갈 수 있습니다.

## B-Tree 인덱스 구조

관계형 DB에서 가장 널리 쓰이는 인덱스는 **B-Tree(정확히는 B+Tree)** 구조입니다.

![B-Tree 인덱스 구조](../../../../Attached%20file/db_btree_index.svg)

- **루트(Root) → 브랜치(Branch) → 리프(Leaf)** 의 계층 트리로 이루어집니다.
- 값을 찾을 때는 루트에서 시작해 "이 값은 어느 범위에 속하나"를 판단하며 아래로 내려갑니다. 이를 **수직적 탐색**이라 하고, 트리 높이만큼만 읽으면 되므로 매우 빠릅니다.
- **리프 노드는 인덱스 키가 정렬된 채 서로 연결(linked list)**되어 있습니다. 그래서 특정 값을 찾은 뒤 옆으로 훑으며(**수평적 탐색**) 범위 검색(`BETWEEN`, 부등호)을 효율적으로 처리할 수 있습니다.
- 리프 노드는 실제 행의 위치인 **ROWID**를 가지고 있어, 이를 통해 테이블의 행에 접근합니다.

## 인덱스 튜닝의 두 가지 핵심 요소

인덱스는 큰 테이블에서 소량의 데이터를 검색할 때 사용합니다. **온라인 트랜잭션 처리(OLTP)** 시스템에서는 소량 데이터를 주로 검색하므로 인덱스 튜닝이 무엇보다 중요합니다. 세부 방법은 다양하지만, 핵심은 크게 두 가지입니다.

### 1. 인덱스 스캔 효율화 튜닝

인덱스를 스캔하는 과정에서 발생하는 비효율을 줄이는 것입니다. 예를 들어 학생명부에서 시력이 1.0~1.5인 홍길동 학생을 찾을 때, 인덱스를 **`이름 + 시력` 순으로 정렬**한 경우와 **`시력 + 이름` 순으로 정렬**한 경우는 읽어야 하는 인덱스 범위가 크게 달라집니다. 즉 **복합 인덱스의 컬럼 순서**가 스캔 효율을 좌우합니다.

### 2. 랜덤 액세스 최소화 튜닝

인덱스 스캔 후 테이블의 레코드에 접근할 때는 **랜덤 I/O** 방식을 사용합니다(리프의 ROWID로 흩어진 행을 하나씩 찾아감). 이 테이블 액세스 횟수를 줄이는 것이 **랜덤 액세스 최소화 튜닝**입니다.

## SQL 튜닝은 랜덤 I/O와의 전쟁

데이터베이스가 느린 근본 이유는 **디스크 I/O** 때문입니다. 읽어야 할 데이터량이 많고 그 과정에 디스크 I/O가 많이 발생할수록 느려집니다. 특히 인덱스를 많이 쓰는 OLTP 시스템에서는 디스크 I/O 중에서도 **랜덤 I/O**가 성능을 좌우합니다.

DBMS가 제공하는 많은 기능이 이 느린 랜덤 I/O를 극복하기 위해 개발되었습니다. 대량 데이터를 조인할 때 쓰는 **소트 머지 조인**과 **해시 조인** 역시 랜덤 I/O를 피하기 위해 고안된 방식입니다. 그만큼 랜덤 I/O를 줄이는 것이 튜닝의 핵심입니다.
