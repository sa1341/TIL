# DB는 왜 데이터보다 로그를 먼저 쓸까 (WAL)

DB에 `UPDATE`를 날리면 데이터를 바로 바꾸지 않습니다. 먼저 **"이런 변경을 할 예정이다"라는 로그를 디스크에 쓰고**, 그다음에 실제 데이터를 바꿉니다. 이것이 **WAL(Write-Ahead Logging)**입니다.

한 번 쓰면 될 걸 왜 두 번 쓸까요? 오히려 느려지는 것 아닐까요?

## 이 설계가 없으면 — Partial Write

DB가 데이터를 **직접 디스크에 쓰는 도중에 서버가 꺼졌다**고 해 봅시다.

- 디스크의 한 페이지는 보통 **8KB**입니다.
- OS는 이걸 한 번에 **원자적으로 쓰지 못합니다.**
- 쓰는 도중 꺼지면 **앞부분은 새 데이터, 뒷부분은 옛 데이터**인 **파셜 라이트(partial write)** 상태가 됩니다.
- 이건 복구할 수 없습니다.

## WAL이 이 문제를 해결한다

WAL은 **append-only 순차 쓰기**입니다.

- 기존 데이터를 덮어쓰지 않고 **끝에 이어붙이기만** 합니다.
- 크래시가 나도, **WAL의 마지막 완전한 레코드까지가 커밋된 데이터**이고, 불완전한 꼬리는 버리면 됩니다.

## WAL도 결국 디스크 쓰기 — 그룹 커밋

그런데 WAL도 결국 디스크 쓰기입니다. 매 커밋마다 **`fsync`**를 호출해서 WAL을 디스크에 확정하는데, 이게 느립니다.

그래서 대부분의 DB는 **여러 트랜잭션의 WAL을 모아서 한 번에 `fsync`하는 그룹 커밋(Group Commit)**을 씁니다.

- 예: 1ms 동안 쌓인 10개의 커밋을 **`fsync` 한 번으로** 처리합니다.

## 정리

> WAL이 데이터보다 먼저 쓰이는 건 **"두 번 써서 느려지는 것"이 아니라, "안전하게 한 번 쓰고 나중에 여유 있게 반영하는 것"**입니다. 순차 쓰기라 오히려 빠르고, 크래시에도 안전합니다.
>
> **빠른 DB는 데이터를 빨리 쓰는 DB가 아니라, 로그를 잘 쓰는 DB입니다.**
