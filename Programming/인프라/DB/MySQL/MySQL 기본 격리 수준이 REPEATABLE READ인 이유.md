# MySQL 기본 격리 수준이 REPEATABLE READ인 이유

DB 격리 수준은 평소에 잘 건드리지 않는 영역입니다. 대개 기본값 그대로 쓰죠. 그런데 다른 DB와 비교해 보면 이상한 점이 하나 있습니다. **거의 모든 DB가 `READ COMMITTED`를 기본으로 쓰는데, MySQL(InnoDB)만 한 단계 더 강한 `REPEATABLE READ`를 기본으로 잡고 있습니다.**

왜 MySQL만 다를까요? 이유는 기술적 우월성이 아니라 **MySQL의 복제(replication) 방식이 남긴 역사**에 있습니다.

## 1. 출발점: 문장 기반 복제(SBR)

MySQL은 초창기부터 **문장 기반 바이너리 로그(Statement-Based Replication, SBR)**로 복제를 했습니다.

- 마스터에서 실행한 변경 SQL을 **그대로** 슬레이브에서 다시 실행하는 방식입니다.
- `INSERT`, `UPDATE` 같은 변경 쿼리를 바이너리 로그(binlog)에 적고, 슬레이브가 그걸 가져가 **똑같이 재실행**합니다.

## 2. READ COMMITTED에서 터지는 문제

문제는 격리 수준이 `READ COMMITTED`일 때 발생합니다. 이 수준에서는 다른 트랜잭션이 **중간에 끼어들 수 있기** 때문입니다.

- 마스터에서는 트랜잭션이 시작될 때 어떤 행(예: `id=1500`)이 아직 없어서 업데이트 대상이 아니었습니다.
- 그런데 슬레이브는 binlog에 기록된 **커밋 순서**대로 재실행합니다.
- 만약 `트랜잭션 2`의 `INSERT`가 먼저 들어간 뒤 `트랜잭션 1`의 `UPDATE`가 실행되면, 슬레이브에서는 1500도 업데이트되어 버립니다.

결과적으로 **마스터와 슬레이브의 데이터가 어긋나는** 복제 정합성 붕괴가 일어납니다.

## 3. REPEATABLE READ가 이걸 막는다

`REPEATABLE READ`는 이 문제를 해결합니다.

- 한 트랜잭션이 처음 본 **스냅샷을 끝까지 유지**합니다.
- InnoDB는 여기에 **갭 락(gap lock)**까지 더해서, 다른 트랜잭션의 **INSERT 자체를 막아** 버립니다.

즉, 중간에 끼어드는 INSERT가 없으니 마스터와 슬레이브의 재실행 결과가 항상 일치합니다. 이렇게 **문장 기반 복제라는 제약이 격리 수준의 기본값까지 정해 버린** 것입니다.

## 4. 반전: 이제는 필요 없어졌다

그런데 정작 요즘 MySQL은 **행 기반 바이너리 로그(Row-Based Replication, RBR)**가 기본입니다.

- RBR은 "어느 행이 어떻게 바뀌었다"는 **변경 결과 자체**를 기록합니다.
- 쿼리를 다시 실행하지 않으므로, 격리 수준이 약해도 복제 정합성이 유지됩니다.
- 즉, **SBR 시절의 제약은 이미 사라졌습니다.**

## 5. 그런데도 기본값은 여전히 REPEATABLE READ

제약이 사라졌는데도 기본값은 바뀌지 않았습니다. 이유는 **호환성**입니다.

- 수많은 운영 시스템이 이미 `REPEATABLE READ`의 동작에 의존하고 있어서, 기본값을 바꾸면 기존 시스템의 동작이 깨질 수 있습니다.
- 오히려 `REPEATABLE READ` + 갭 락은 **동시 INSERT가 많은 워크로드에서 락 컨텐션(lock contention)과 데드락의 원인**이 되기도 합니다.
- 그래서 결제나 좌석 예매처럼 의도치 않은 락 대기가 치명적인 일부 운영 시스템에서는, **일부러 `READ COMMITTED`로 낮춰서** 쓰기도 합니다.

## 정리

> MySQL의 기본 격리 수준이 `REPEATABLE READ`인 것은 **기술적 합의가 아니라 역사적 부채**입니다. 문장 기반 복제 시절의 정합성 요구가 기본값을 정했고, 복제 방식이 행 기반으로 바뀐 지금도 호환성 때문에 그대로 남아 있습니다.
