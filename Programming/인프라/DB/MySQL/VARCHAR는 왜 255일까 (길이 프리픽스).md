# VARCHAR는 왜 255일까 (길이 프리픽스)

DB 스키마에서 자주 보이는 숫자가 있습니다. `VARCHAR(255)`. 그런데 다른 누군가의 스키마에선 `VARCHAR(85)`, `VARCHAR(63)`이 보일 때도 있습니다. 256도 128도 64도 아닌 어정쩡한 이 세 숫자 — 사실은 **모두 같은 "1바이트의 한계"에서 나온 숫자**입니다. 단, 어떤 문자셋을 쓰느냐에 따라 정답이 달라지죠.

## 길이 프리픽스가 핵심

MySQL InnoDB는 VARCHAR 데이터 앞에 **길이 프리픽스(length prefix)**를 붙입니다.

- 프리픽스가 **1바이트**면 0~255까지 표현 가능합니다.
- 그런데 `VARCHAR(N)`의 **N은 바이트 수가 아니라 문자 수**입니다.
- 그래서 **"문자셋의 한 글자 최대 바이트"**가 핵심이 됩니다.

문자셋별로 1바이트 프리픽스(≤255바이트)에 들어가는 최대 문자 수:

| 문자셋 | 글자당 바이트 | 1바이트 프리픽스 경계 |
|---|---|---|
| latin1 | 1 | **255** (255×1=255) |
| utf8 (mb3) | 3 | **85** (85×3=255) |
| utf8mb4 | 4 | **63** (63×4=252 ≤ 255, 64×4=256 초과) |

즉 `VARCHAR(255)`는 **latin1 시절의 최적값**이었습니다.

## 왜 경계를 넘으면 손해인가

utf8mb4 기준 `VARCHAR(64)`부터는 `64 × 4 = 256`바이트라 **프리픽스가 2바이트로 늘어납니다.**

- 행이 1억 개면 단순 계산으로 **100MB가 그냥 추가**됩니다.
- 인덱스에도 프리픽스가 포함되니, **B-Tree 페이지에 들어가는 키 수가 줄어** 페이지 스플릿이 더 자주 일어납니다.

## 모든 DB가 이렇진 않다

- **PostgreSQL**은 가변 길이 구조(**varlena**)라, MySQL 같은 255바이트 경계 자체가 없습니다. 255든 256이든 저장 비용 차이가 없죠.
- 그리고 MySQL이 **utf8mb4를 기본으로 쓰는 시대**에 무지성으로 `VARCHAR(255)`를 박는 건, 사실 **"내 컬럼은 길이 프리픽스가 2바이트입니다"라고 선언하는 셈**입니다. 진짜 1바이트 프리픽스를 노렸다면 `VARCHAR(63)`이 정확한 답이죠. (물론 80자짜리 데이터를 63으로 자르는 건 본말전도입니다.)

## 정리

> `VARCHAR(255)`는 **latin1 시절의 최적값**이었을 뿐, utf8mb4 시대의 정답은 **데이터 도메인이 결정하는 것이지 관습이 결정하는 게 아닙니다.**
