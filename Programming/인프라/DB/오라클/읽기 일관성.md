## 1. SCN이란? (System Change Number)

**SCN = System Change Number.** 데이터베이스에 일어나는 모든 커밋에 순차적으로 부여되는 **논리적 타임스탬프**다. 실제 시계 시간이 아니라 Oracle 내부에서 관리하는 단조증가(monotonically increasing) 카운터다.

핵심 성질은 딱 하나만 기억하면 된다: **SCN은 커밋(commit)할 때만 증가한다.** 조회(SELECT)만으로는 절대 오르지 않는다.

![SCN은 커밋할 때만 증가](img/01_scn_basics.svg)

그래서 트랜잭션 A가 조회할 때 SCN이 100이고, 그 사이 아무도 커밋하지 않았다면, 다음에 조회하는 트랜잭션 B도 여전히 100을 본다. "시간이 흐르면 자동으로 똑딱거리는 시계"가 아니라 **"커밋 이벤트에만 반응하는 카운터"**라는 게 포인트.

### SCN이 쓰이는 곳
- **읽기 일관성 (Read Consistency)** — 이 문서의 주제
- **Crash Recovery** — 컨트롤 파일 / redo 로그 / 데이터파일 헤더의 SCN 비교로 복구 지점 결정
- **RMAN 백업 / Data Guard** — 백업·복제 시점을 SCN으로 태깅·동기화
- **Flashback Query** — `SELECT ... AS OF SCN` 으로 과거 시점 조회

### MySQL(InnoDB)의 대응 개념 = `trx_id`
MySQL에는 SCN이라는 이름은 없지만 개념적으로 거의 1:1로 대응하는 게 **Transaction ID(`trx_id`)**다.

| Oracle | MySQL(InnoDB) |
|---|---|
| SCN | `trx_id` (트랜잭션 ID) |
| SCN 기반 읽기 일관성 | `trx_id` + `Read View` 기반 MVCC |
| UNDO Segment로 과거 이미지 복원 | undo log(rollback segment) 체인으로 복원 |
| 각 row에 마지막 변경 SCN | 각 row 숨김 컬럼 `DB_TRX_ID` + `DB_ROLL_PTR` |

`trx_id`도 **쓰기 트랜잭션에만 할당**되고, 순수 읽기 전용 트랜잭션은 아예 안 받기도 한다. 그래서 "읽기만으로는 시점 카운터가 안 움직인다"는 원리가 양쪽 모두 동일하게 적용된다.

## 2. READ COMMITTED — SCN은 '문장' 단위로 새로 찍힌다

Oracle의 **기본 격리수준은 READ COMMITTED**다. 여기서 스냅샷 SCN이 고정되는 시점은 **트랜잭션 시작이 아니라 각 SQL 문장(statement)이 열리는 시점**이다.

> 공식 문서 표현: READ COMMITTED에서 일관성 기준점은 **"the time at which the statement was opened"**. 문장이 SCN 1000에 열리면 그 문장은 SCN 1000에 일관.

![READ COMMITTED는 문장 단위 스냅샷](img/02_read_committed.svg)

그래서 **한 트랜잭션 안에서 같은 쿼리를 두 번 실행**하고 그 사이에 다른 트랜잭션이 커밋했다면:

- 1차 문장 → 시작 시점 SCN 100 기준
- (그 사이 T2가 UPDATE + COMMIT → SCN 101)
- 2차 문장 → **새로운 문장이므로 시작 시점 SCN 101을 다시 찍음** → 바뀐 값이 보임

결과가 달라진다. 이게 바로 **Non-repeatable Read**이고, READ COMMITTED에서는 정상 동작이다.

### MySQL REPEATABLE READ와의 결정적 차이
MySQL InnoDB의 기본 격리수준인 REPEATABLE READ는 **트랜잭션 시작 시 Read View를 딱 한 번만 고정**한다. 그래서 같은 트랜잭션 안에서 몇 번을 조회하든 계속 같은 시점을 본다 → Non-repeatable Read가 안 생긴다.

| | Oracle READ COMMITTED | MySQL REPEATABLE READ |
|---|---|---|
| 스냅샷 고정 시점 | **매 SQL 문장마다** | **트랜잭션 시작 시 1번** |
| 같은 TX 내 반복 조회 | 결과 달라질 수 있음 | 항상 같음 |
| Non-repeatable Read | 발생 (정상) | 방지됨 |

## 3. 핵심 — 왜 '옛날' 데이터를 보는가? (UNDO / CR Clone)

여기가 제일 헷갈리는 지점. **"문장이 시작된 이후"**와 **"문장이 실행되는 도중"**을 구분해야 한다.

READ COMMITTED가 보장하는 건 **"문장이 시작한 시점에 커밋되어 있던 데이터"**이지, "실행 도중에 방금 커밋된 걸 실시간으로 따라잡아 보여준다"가 아니다.

![UNDO로 CR Clone 복원](img/03_cr_clone_undo.svg)

### 왜 실시간으로 최신을 보면 안 되나 (모순 발생)
100만 건짜리 `SELECT`가 5초 걸린다고 하자. 실행 도중 다른 트랜잭션이 row 하나를 바꿔 커밋했을 때, 만약 그 최신값을 그대로 보여주면:

- row 1~49 → 옛날(SCN 100) 값
- row 50 → 방금 바뀐(SCN 101) 값
- row 51~100 → 다시 옛날(SCN 100) 값

**한 번의 SELECT 결과 안에 서로 다른 시점의 데이터가 섞인다**(fuzzy read). 논리적으로 "지금 이 순간의 스냅샷"이라 볼 수 없는 뒤죽박죽 상태다. Oracle은 이걸 허용하지 않는다.

### 그래서 하는 일: CR Clone
Oracle은 블록을 **실제로 읽는 순간** "이 블록의 SCN vs 내 문장 시작 SCN"을 비교한다.

1. 블록 헤더의 **ITL(Interested Transaction List)** 로 "이 row를 누가 언제(어느 SCN에) 바꿨는지" 확인
2. 내 쿼리 SCN(100) < 블록 SCN(101) 이면 → 예전 이미지 필요
3. row의 UNDO 포인터를 따라 **UNDO Segment**에서 변경 직전 값(before-image)을 찾음
4. UNDO 레코드의 SCN이 내 쿼리 SCN 이하가 될 때까지 체인을 거슬러 올라감
5. 복원한 과거 이미지를 **별도 버퍼에 CR Clone(Consistent Read Clone)** 으로 만들어 결과로 반환

**원본 블록(buffer cache)은 최신 상태 그대로 유지**되고, 복원본만 쿼리에게 돌려준다. 이렇게 해서 한 문장의 모든 row가 **동일한 SCN 시점**으로 통일된다(= Statement-level Read Consistency).

### 두 케이스를 확실히 구분

| 상황 | SCN 기준 | UNDO 역할 |
|---|---|---|
| **한 문장 실행 도중** 다른 TX가 커밋 | 문장 시작 시점(예: 100) 유지 | 그 이후 변경분을 UNDO로 가려 **과거 이미지 복원** |
| **문장 A 종료 후, 문장 B 시작 전**에 다른 TX가 커밋 | 문장 B는 **새 시점(101)** 을 다시 찍음 | 복원 없이 **최신값** 그대로 (Non-repeatable Read) |

> ⚠ **ORA-01555 "snapshot too old"**: 위 3번에서 필요한 과거 UNDO 이미지가 이미 다른 트랜잭션에 의해 재사용(overwrite)돼 복원할 수 없을 때 발생. 오래 걸리는 배치 쿼리 + 작은 `UNDO_RETENTION` / UNDO 테이블스페이스 부족이 전형적인 원인.

## 4. JDBC — `fetchSize`와 `executeQuery` / `next()` 동작

`ResultSet.next()`는 대부분 **드라이버 로컬 버퍼(메모리)에서 1건씩 꺼내는 것**이고, 버퍼가 비었을 때만 서버 왕복이 일어난다. 이 동작이 Oracle과 MySQL이 완전히 다르다.

![JDBC fetch 동작 Oracle vs MySQL](img/04_jdbc_fetch.svg)

### Oracle JDBC
- `executeQuery()` → 서버에 **커서 open**(이 시점에 문장 시작 SCN 고정), execute와 첫 fetch를 묶어서 왕복 1회로 처리
- `next()`가 버퍼를 소진하면 → **fetchSize 단위로 서버에 재요청**(커서는 열린 채 유지)
- 기본 fetchSize = **10** → 보수적이라 조건 없는 대량 조회에도 OOM은 잘 안 남. 다만 너무 작아서 왕복이 많아지므로 실무에선 100~500으로 **올려서** 튜닝
- **23ai 드라이버는 adaptive fetch size** — row 크기에 따라 4번째 fetch에서 최대 250까지 자동 증가. 최신 드라이버를 쓰면 fetchSize가 상당 부분 알아서 조정됨
- 전역 기본값은 `defaultRowPrefetch` 커넥션 속성으로 변경 가능

### MySQL Connector/J — 기본 모드 (⚠ 함정)
- 공식 문서: **"By default, ResultSets are completely retrieved and stored in memory."**
- `executeQuery()` 시점에 **전체 결과셋을 JVM Heap에 통째로 로드** → 조건 없는 100만 건이면 실제로 **OOM**
- 이후 `next()`는 이미 로컬 메모리에 있는 데이터를 순회할 뿐(서버 왕복 없음), 일반 양수 fetchSize는 사실상 무의미

### MySQL — 안전하게 하는 두 가지 방법

**A. Row-by-row 스트리밍**
```java
stmt = conn.createStatement(
    ResultSet.TYPE_FORWARD_ONLY, ResultSet.CONCUR_READ_ONLY);
stmt.setFetchSize(Integer.MIN_VALUE);   // 특수 매직값
```
forward-only + read-only 조합이 스트리밍 신호. `next()`마다 소켓에서 1 row씩 읽는다(버퍼링 X).
주의: 다 읽거나 닫기 전에는 그 커넥션으로 다른 쿼리를 실행할 수 없다.

**B. 서버 커서 (Oracle 방식에 가장 근접)**
```java
// jdbc:mysql://host/db?useCursorFetch=true
stmt.setFetchSize(1000);   // N건씩 서버에서 나눠 받음
```
N건씩 서버 커서로 나눠 받는다. Oracle의 fetchSize와 가장 비슷하게 동작하므로, **배치 처리 시 깔끔한 선택지**다.

> 참고: `Integer.MIN_VALUE` 매직값은 안정적이지만 커넥션 점유 제약이 있어, 대량 배치에서는 보통 **B(useCursorFetch) 방식**을 더 선호한다.

## 5. 한눈에 보기 — Oracle vs MySQL

| 항목 | Oracle | MySQL(InnoDB) |
|---|---|---|
| 시점 식별자 | SCN (커밋마다 +1) | `trx_id` |
| 기본 격리수준 | **READ COMMITTED** | **REPEATABLE READ** |
| 스냅샷 고정 | 문장(statement)마다 | 트랜잭션 시작 시 1번 |
| 과거 이미지 저장소 | UNDO Segment | undo log (rollback segment) |
| 복원 결과물 | CR Clone (별도 버퍼) | undo chain으로 재구성한 버전 |
| 오래된 스냅샷 에러 | ORA-01555 (snapshot too old) | (거의 없음; purge 지연 시 history list 증가) |
| JDBC 기본 fetch | 10건씩 서버 왕복 (23ai adaptive) | **전체를 메모리로 로드** |
| 대량 조회 안전장치 | 기본값이 보수적 → 대체로 안전 | 명시적 스트리밍/서버커서 필요 |

## 6. 헷갈렸던 포인트 요약 (Q&A)

**Q. SCN은 조회할 때도 변하나?**
아니다. **커밋할 때만** 증가한다. A가 조회 시 100이고 그 사이 커밋이 없으면 B도 100.

**Q. READ COMMITTED니까 다시 읽으면 최신(101)을 보나?**
그렇다. **새 문장**은 시작 시점의 최신 SCN을 다시 찍는다. 단 이건 "문장 사이"의 이야기다.

**Q. 그런데 왜 UNDO로 옛날 걸 복원한다는 거야?**
그건 **한 문장이 실행되는 도중**의 이야기다. 문장 실행 중 다른 TX가 커밋해도, 그 문장은 시작 시점 SCN으로 통일해서 봐야 결과가 뒤섞이지 않기 때문(fuzzy read 방지). 이때만 UNDO로 과거 이미지를 복원한다.

**Q. fetch 도중(예: 51~100 fetch 중) 53번 row가 바뀌면?**
문장 시작 SCN(100) 기준 UNDO 이미지로 가져온다. **Oracle도 그렇다.** 단, 이미 fetch돼 넘어간 앞쪽 row는 복원 대상이 아니다(이미 확정된 결과).

**Q. 조건 없는 100만 건 조회, fetchSize 안 걸면 OOM 나나?**
- Oracle: 기본 10이라 대체로 안전(왕복만 많아짐)
- MySQL: 기본이 **전체 메모리 로드**라 진짜 OOM 위험 → 스트리밍/`useCursorFetch` 꼭 켜야 함
