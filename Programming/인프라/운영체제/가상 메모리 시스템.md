# 가상 메모리 시스템

## 한 줄 요약

> **가상 메모리 = "프로세스에게는 넉넉하고 연속된 나만의 메모리가 있는 것처럼 보여주고, 실제로는 물리 메모리(RAM) + 디스크(스왑)를 조합해서 그 환상을 만들어 주는 기술"**

## 가상 메모리가 왜 필요한가?

컴퓨터마다 물리 메모리(RAM) 크기가 다르고, 여러 프로그램이 동시에 돌아갑니다. 만약 프로그램이 "물리 메모리의 몇 번지에 내가 올라갈지"를 직접 신경 써야 한다면, 컴퓨터가 바뀔 때마다, 또 다른 프로그램과 자리가 겹칠 때마다 코드를 고쳐야 할 것입니다.

가상 메모리는 이 문제를 없애 줍니다. 모든 프로세스는 물리 메모리의 실제 위치나 크기와 **무관하게**, 자신만의 **0번지부터 시작하는 연속된 주소 공간**을 가진 것처럼 프로그래밍할 수 있습니다. "내 메모리는 나 혼자 통째로 쓴다"고 가정하면 되는 것이죠.

## 가상 주소 공간의 크기 — 무엇이 결정하는가?

가상 주소 공간의 최대 크기는 **물리 메모리(RAM) 크기가 아니라 CPU의 주소 비트 수**가 결정합니다.

- 32bit CPU → 주소를 2³² = **4GB**까지 표현 가능 → 프로세스 하나당 가상 주소 공간 최대 4GB
- 64bit CPU → 이론상 2⁶⁴ (실제로는 48bit 등으로 제한, 그래도 수백 TB 이상)

여기서 핵심은, **물리 메모리가 4GB보다 작아도 각 프로세스는 4GB 주소 공간을 가진 것처럼 동작한다**는 점입니다. 즉 가상 주소 공간의 크기와 실제 RAM 크기는 별개입니다.

> ⚠️ 흔한 오해: "물리 메모리가 4GB라서 가상 메모리도 4GB가 한계다"는 틀립니다. 가상 주소 공간의 크기는 주소 비트 수로 정해지고, **실제로 담을 수 있는 데이터 총량**은 `물리 메모리 + 스왑 영역`입니다.

## 어떻게 "무한대처럼" 동작하는가? — 스왑

4GB 주소 공간을 쓰는 프로세스 10개를 동시에 실행하려면 단순 계산으로는 40GB가 넘는 메모리가 필요합니다. 하지만 실제 RAM은 그보다 훨씬 작을 수 있죠. 어떻게 가능할까요?

비결은 **당장 쓰지 않는 부분은 RAM에 올려두지 않는다**는 데 있습니다. 프로세스가 지금 이 순간 실제로 건드리는 메모리는 전체의 일부분뿐이라서, 그 일부만 RAM에 두고 나머지는 디스크에 미뤄 둡니다.

- **스왑 영역(swap area)**: 디스크에 있지만 메모리 관리자가 "물리 메모리의 연장"처럼 관리하는 공간. 가상 메모리의 구성 요소 중 하나입니다.
- **스왑아웃(swap-out)**: RAM이 꽉 차면, 당분간 안 쓸 것 같은 페이지를 스왑 영역으로 내보냅니다.
- **스왑인(swap-in)**: 내보냈던 페이지가 다시 필요해지면 스왑 영역에서 RAM으로 가져옵니다.

> 메모리 관리자가 쓸 수 있는 전체 메모리 크기 = **물리 메모리 + 스왑 영역**

## 동적 주소 변환 (Dynamic Address Translation)

프로세스는 가상 주소만 사용하지만, CPU가 실제로 데이터를 읽으려면 물리 주소가 필요합니다. 그래서 **가상 주소 → 물리 주소** 변환이 매 접근마다 일어나는데, 이를 **동적 주소 변환**이라 하고 하드웨어인 **MMU(Memory Management Unit)**가 담당합니다.

이 변환의 기준표가 **페이지 테이블(page table)**입니다.

1. 가상 주소 공간을 **페이지(page)**라는 고정 크기 조각(보통 4KB)으로 나눕니다.
2. 물리 메모리도 같은 크기의 **프레임(frame)**으로 나눕니다.
3. 페이지 테이블이 "가상 페이지 N → 물리 프레임 M" 매핑을 기록합니다.
4. MMU는 이 표를 보고 가상 주소를 물리 주소로 즉시 변환합니다.

덕분에 프로세스는 물리 메모리의 어디에, 어떤 순서로 배치되는지 전혀 신경 쓰지 않고 데이터를 마음대로 쓸 수 있습니다.

## 페이지 폴트 (Page Fault)

접근하려는 페이지가 **물리 메모리에 없고 스왑 영역에 있을 때** 발생하는 이벤트입니다.

1. MMU가 페이지 테이블을 보니 해당 페이지가 RAM에 없음(=페이지 폴트) → 운영체제에 알림
2. 운영체제가 스왑 영역에서 그 페이지를 찾아 RAM의 빈 프레임으로 **스왑인**
3. (빈 프레임이 없으면 교체 알고리즘(LRU 등)으로 기존 페이지를 하나 **스왑아웃**)
4. 페이지 테이블을 갱신하고, 중단됐던 접근을 다시 실행

이렇게 "필요할 때 그때그때 올리는" 방식을 **요구 페이징(demand paging)**이라고 합니다. 프로그램 전체를 미리 올리지 않고, 실제로 접근되는 페이지만 올려서 메모리를 아낍니다.

> 페이지 폴트가 너무 자주 일어나 스왑인/스왑아웃만 반복하며 실제 일을 못 하는 상태를 **스래싱(thrashing)**이라고 합니다. RAM이 부족할 때 시스템이 급격히 느려지는 대표적인 원인입니다.

## 정리

- 프로세스는 **0번지부터의 연속된 가상 주소 공간**만 보면 된다 → 프로그래밍이 단순해짐
- 가상 주소 공간 크기는 **CPU 주소 비트 수**가 결정 (32bit=4GB), 실제 담을 수 있는 양은 **RAM + 스왑**
- **MMU + 페이지 테이블**이 가상 주소를 물리 주소로 변환 (동적 주소 변환)
- 당장 필요한 페이지만 RAM에 올리고(요구 페이징), 부족하면 디스크(스왑)로 밀어낸다
