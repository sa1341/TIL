# 새벽에만 502가 뜨는 이유 (로드밸런서와 서버의 타임아웃 불일치)

새벽에 **502 에러**가 몇 건 터졌습니다. 그런데 서버에 들어가 보면 아무 이상이 없습니다. CPU도 낮고, 메모리도 여유롭고, 로그에도 에러가 없습니다. 그리고 **낮에는 절대 안 터집니다.** 원인은 서버가 아니라 **로드밸런서(LB)와 서버 사이의 타이밍**이었습니다.

## Keep-Alive 커넥션의 재사용

요청이 들어오면 보통 **LB를 거쳐** 서버에 도달합니다. LB와 서버 사이에는 커넥션이 하나 만들어지고, 이 커넥션은 요청이 끝나도 바로 끊지 않습니다. 다음 요청이 오면 **재사용하려고 열어 둡니다.** 이게 **Keep-Alive**입니다.

문제는 **이 커넥션을 얼마나 오래 열어 둘지 LB와 서버가 따로 정한다**는 것입니다.

## 경쟁 상태(Race Condition)

- AWS ALB의 기본 idle timeout은 **60초**입니다.
- 그런데 서버 쪽 Keep-Alive 타임아웃이 LB보다 **짧으면** 문제가 생깁니다. (예: gunicorn 기본 2초, Node.js 5초)
- 서버가 2초 만에 커넥션을 끊는데, **LB는 이 커넥션이 아직 60초까지 살아 있다고 생각**합니다.
- 이 틈에 새 요청이 들어오면, LB는 **이미 서버가 끊은 커넥션에 요청을 보냅니다.** → 502.

정확히는 **서버가 끊는 순간과 LB가 요청을 보내는 순간이 거의 동시에 일어나는 경쟁 상태**입니다.

**새벽에만** 발생하는 이유는, 트래픽이 적어서 커넥션이 유휴 상태로 오래 남기 때문입니다. 낮에는 요청이 계속 오니 서버 타임아웃 안에 커넥션이 재사용되어 끊길 일이 없습니다.

## 해결 — 서버 타임아웃을 LB보다 길게

해결은 간단합니다. **서버의 Keep-Alive 타임아웃을 LB보다 길게** 맞추면 됩니다. LB가 **항상 먼저 끊게** 만들면, 서버가 끊은 커넥션에 요청을 보내는 일이 사라집니다. (AWS 공식 권장도 동일합니다.)

## NLB로 바꾸면 다시 터진다

NLB는 **L4 레벨**이라 idle timeout이 **350초**입니다. 서버 Keep-Alive를 65초로 맞춰 놨는데 NLB로 전환하면, 서버는 65초에 끊는데 NLB는 350초까지 유효하다고 생각합니다. **같은 문제가 다시 터집니다.**

## 정리

> 간헐적 502의 원인은 **서버 코드가 아니라 LB와 서버 사이의 타이머 불일치**입니다. 서버만 보면 절대 못 찾습니다. 항상 **LB가 먼저 커넥션을 끊도록** 타임아웃을 정렬하세요.
