# HTTP와 TCP/IP의 차이점

업무를 하면서 HTTP 통신으로 데이터를 주고받는 RESTful API를 자주 작성하는데, 문득 TCP/IP와 HTTP의 차이점이 무엇인지 생각이 나서 구글링을 해보았습니다. 역시 웹상에서 빡고수분들이 너무 잘 정리해 주어서, 이해가 잘 되도록 정리해 보았습니다.

일단 대학교 시절에 OSI 7계층에 대해서는 동해물과 백두산이 마르도록 들었고, 시험을 위해 외웠던 기억이 있습니다. 요즘은 웹 서비스에 맞게 단순화한 **TCP/IP 4계층**을 주로 사용합니다. 여기서 **TCP는 전송 계층, IP는 인터넷 계층**에 속합니다.

## TCP/IP 4계층 구조

![TCP/IP 4계층과 캡슐화](../../../Attached%20file/net_tcpip_layers.svg)

각 계층은 자기 역할만 책임지고, 아래 계층에게 전달을 위임합니다.

- **애플리케이션 계층**: HTTP, DNS, FTP 등. 사람이 쓰는 서비스 수준의 데이터를 다룹니다. HTTP가 여기서 동작합니다.
- **전송 계층**: TCP, UDP. **포트 번호**로 어떤 애플리케이션에 전달할지 구분하고, TCP는 신뢰성과 순서를 보장합니다.
- **인터넷 계층**: IP. **IP 주소**로 목적지까지의 경로(라우팅)를 찾습니다.
- **네트워크 접근 계층**: Ethernet, Wi-Fi. **MAC 주소**를 이용해 실제 물리 매체로 비트를 전송합니다.

즉 HTTP는 최상위 계층에서 동작하며, 개념적으로 **HTTP는 TCP/IP 계층 위에 얹혀서 동작**한다고 볼 수 있습니다.

> 참고: IP 계층에는 출발지·목적지 **주소(IP)**가, TCP 계층에는 출발지·목적지 **포트**가 있습니다. IP가 아파트(건물) 주소라면 TCP 포트는 동·호수 개념이라고 생각하면 편합니다. 둘이 합쳐져야 정확한 목적지에 도착합니다.

## HTTP vs TCP — 무엇이 다른가

| 구분 | TCP | HTTP |
|---|---|---|
| 계층 | 전송 계층 | 애플리케이션 계층 |
| 데이터 형태 | 바이트(byte) 스트림 | 문자열(String) 메시지 |
| 역할 | 신뢰성 있는 전송 통로 | 그 통로 위에서 주고받는 규약 |

## 요청이 흘러가는 과정

예를 들어 클라이언트가 특정 URL로 요청을 보내면 다음 순서로 처리됩니다.

1. **DNS**가 도메인에 매핑되는 IP 주소를 찾아옵니다.
2. **전송 계층(TCP)**에서 HTTP 메시지를 세그먼트로 나눕니다.
3. **인터넷 계층(IP)**에서 목적지까지의 경로를 확인하고,
4. **네트워크 접근 계층**을 통해 실제로 전송합니다.

받는 쪽은 이 과정을 **역순**으로 진행하여(역캡슐화) 원래의 HTTP 메시지를 복원합니다.

> 소켓도 TCP 기반으로 나온 것인데, HTTP와의 차이점은 **연결 지향 / 동기식 통신**이 필요할 때는 소켓 통신을 이용하는 것이 더 유리하다는 점입니다.
