# SSL/TLS 내부 동작: Pre-Master Secret부터 대칭키 파생까지

최근 실무에서 


- **인사 및 탐색 (Hello) 🤝**
    
    - 클라이언트와 서버가 서로 지원하는 TLS 버전과 암호화 방식(Cipher Suite)을 확인합니다.
        
    - 이때 양쪽은 나중에 키를 섞을 때 재료로 쓸 **난수(Client/Server Random)**를 평문으로 교환합니다. 🎲
        
- **신원 확인 (Certificate) 📜**
    
    - 서버는 인증서를 보내 자신의 신원을 증명합니다.
        
    - 클라이언트는 **신뢰 저장소(JVM의 cacerts 등)**를 통해 인증서의 유효성을 검증합니다.
        
- **비밀 재료 생성 (Pre-Master Secret) 🗳️**
    
    - **RSA 방식:** 클라이언트가 비밀 값을 생성해 서버의 공개키로 암호화해서 보냅니다.
        
    - **ECDHE 방식:** 각자 수학적 계산을 통해 동일한 비밀 값을 직접 유도해냅니다. 이 방식은 비밀 값을 전송하지 않아 더 안전합니다. ⛰️
        
- **최종 비밀번호 요리 (Master Secret) 🧑‍🍳**
    
    - 공유된 **Pre-Master Secret**과 처음에 주고받은 **난수들**을 특수 함수(PRF)에 넣어 48바이트의 **마스터 시크릿**을 만듭니다. ✨
        
    - 이 마스터 시크릿은 실제 암호화에 쓰일 모든 키의 '원본 반죽'이 됩니다.
        
- **용도별 대칭키 파생 (KDF & Key Expansion) 🏭**
    
    - 마스터 시크릿을 **키 파생 함수(KDF)**에 넣어 여러 조각으로 쪼갭니다.
        
    - 여기서 **Client/Server Write Key**(실제 데이터 암호화), **MAC Key**(메시지 위변조 확인), **IV**(암호화 보안 강화) 등이 탄생합니다.
        
- **최종 확인 및 종료 (Finished) ✅**
    
    - 지금까지 나눈 대화 기록의 요약본(Hash)을 각자의 대칭키로 암호화해서 주고받습니다.
        
    - 서로의 요약본이 일치하면 보안 통로가 완성되어 핸드셰이크가 끝납니다.
