# HTTP/2 맛보기

HTTP/1.1 버전이 나온 지 벌써 20년이 넘었습니다.

현재의 웹은 멀티미디어 리소스를 처리해야 하고, 웹 페이지 하나를 구성하기 위해 다수의 비동기 요청이 발생하고 있는데, 이를 처리하기엔 HTTP/1.1 스펙은 너무 느리고 비효율적입니다. HTTP/2는 이 문제를 해결하기 위해 2015년에 표준화되었습니다.

## HTTP/1.1 동작 방식

HTTP/1.1은 기본적으로 커넥션당 하나의 요청을 처리하도록 설계되었습니다. 그래서 동시 전송이 불가능하고 요청과 응답이 순차적으로 이루어집니다.

그렇다 보니 HTTP 문서 안에 포함된 다수의 리소스를 처리하려면, 요청할 `리소스(html, image, css, js 등)`의 개수에 비례해서 Latency(대기 시간)가 길어질 수밖에 없습니다.

## HOL(Head Of Line) Blocking

웹 환경에서 HOL 블로킹은 실제로 두 종류(HTTP 레벨, TCP 레벨)가 있는데, 여기서는 먼저 HTTP의 HOL 블로킹을 알아봤습니다.

HTTP/1.1의 "커넥션당 하나의 요청 처리"를 개선하는 기법 중 **파이프라이닝(pipelining)**이 있습니다. 하나의 커넥션으로 다수의 파일을 요청/응답 받는 기법인데, 어느 정도 성능 향상을 꾀할 수 있으나 큰 문제점이 있습니다.

하나의 TCP 통신에서 3개의 이미지(a.png, b.png, c.png)를 순차적으로 얻으려는 경우, 첫 번째 이미지인 a.png 응답이 지연되면 **뒤따르는 b.png, c.png의 응답 처리가 a.png가 끝날 때까지 전부 지연**됩니다. 파이프라이닝은 요청 순서대로 응답을 돌려줘야 하기 때문입니다. 이와 같은 현상을 HTTP의 **Head Of Line Blocking**이라 부르며, 파이프라이닝의 큰 한계입니다.

그래서 실무에서는 파이프라이닝 대신 **커넥션을 여러 개 여는 방식**으로 우회했지만, 이는 커넥션 남발과 서버 부하라는 또 다른 비용을 낳습니다.

## HTTP/2의 핵심 개선

HTTP/2는 **하나의 커넥션 위에서 여러 요청을 동시에** 처리할 수 있게 만들어 위 문제를 근본적으로 해결합니다.

### 1. 바이너리 프레이밍 (Binary Framing)

HTTP/1.1은 사람이 읽을 수 있는 **텍스트** 메시지였지만, HTTP/2는 메시지를 **바이너리 프레임(frame)** 단위로 쪼개어 주고받습니다. 파싱이 빠르고 오류가 적으며, 아래의 멀티플렉싱을 가능하게 하는 토대입니다.

- **스트림(Stream)**: 하나의 커넥션 안에서 오가는 하나의 요청/응답 흐름 (고유 ID를 가짐)
- **프레임(Frame)**: 통신의 최소 단위. 여러 스트림의 프레임이 뒤섞여 전송됨

### 2. 멀티플렉싱 (Multiplexing) — HTTP HOL 블로킹 해소

여러 스트림의 프레임을 **하나의 커넥션에서 뒤섞어(interleave)** 동시에 전송합니다. 덕분에 특정 응답(b.png)이 느려도 다른 응답(a.html, c.js)은 막히지 않고 계속 흐릅니다. 커넥션을 여러 개 열 필요도 사라집니다.

### 3. 헤더 압축 (HPACK)

HTTP 요청마다 반복되는 헤더(User-Agent, Cookie 등)를 **HPACK** 알고리즘으로 압축하고, 이전에 보낸 헤더는 인덱스로 참조해 중복 전송을 줄입니다. 요청이 많은 웹 페이지에서 헤더 오버헤드가 크게 감소합니다.

### 4. 스트림 우선순위 & 서버 푸시

- **스트림 우선순위**: 중요한 리소스(예: CSS)를 먼저 받도록 우선순위를 지정할 수 있습니다.
- **서버 푸시(Server Push)**: 클라이언트가 요청하기 전에 서버가 필요한 리소스를 미리 보내는 기능. 다만 실효성 논란으로 **현재는 사실상 폐기**되었습니다(Chrome은 2022년에 지원 제거).

## 그런데 HTTP/2도 완벽하진 않다 — TCP HOL 블로킹

HTTP/2는 **HTTP 레벨**의 HOL 블로킹은 해결했지만, 하나의 TCP 커넥션을 쓰기 때문에 **TCP 레벨**의 HOL 블로킹은 여전히 남습니다. TCP는 순서를 보장하므로, 중간 패킷 하나가 유실되면 그 패킷이 재전송될 때까지 **모든 스트림이 함께 멈춥니다.**

이 문제까지 해결한 것이 **HTTP/3**입니다. HTTP/3는 TCP 대신 UDP 기반의 **QUIC** 프로토콜 위에서 동작하여, 스트림별로 독립적으로 데이터를 처리해 TCP HOL 블로킹까지 제거했습니다.

## 정리

| 구분 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---|---|---|---|
| 전송 형식 | 텍스트 | 바이너리 프레임 | 바이너리 프레임 |
| 동시 처리 | 커넥션당 1요청 | 멀티플렉싱 | 멀티플렉싱 |
| 기반 프로토콜 | TCP | TCP | **UDP(QUIC)** |
| HTTP HOL | 있음 | 해결 | 해결 |
| TCP HOL | 있음 | **여전히 있음** | 해결 |
| 헤더 압축 | 없음 | HPACK | QPACK |
