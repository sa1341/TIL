# API 하위 호환성과 JSON 파싱 전략

여러 클라이언트(안드로이드 두 팀 + 프론트엔드)가 쓰는 API가 있었습니다. 프론트에서만 필요한 필드가 생겨서 하나 추가했습니다. **기존 필드는 그대로 두고 추가만** 했죠. 그런데 배포 후, **한 안드로이드 앱에서 해당 API 호출이 전부 실패**하기 시작했습니다. 같은 안드로이드인데 왜 한쪽만?

![API 하위 호환 — 파싱 라이브러리에 따라 갈리는 결과](../../../Attached%20file/net_api_compat_parsing.svg)

## 원인 — 파싱 라이브러리의 전략 차이

같은 플랫폼이어도 **JSON 파싱 라이브러리마다 전략이 다릅니다.**

- **A팀은 Gson** — 기본적으로 **모르는 필드를 무시**합니다. (프론트엔드 JavaScript도 마찬가지로, 필요한 것만 꺼내니 나머지는 그냥 남습니다.)
- **B팀은 kotlinx.serialization** — 기본 설정이 **엄격 파싱**입니다. 모델에 정의하지 않은 필드가 내려오면 **역직렬화 자체가 실패**합니다.

서버 입장에서는 **필드를 추가한 것뿐**입니다. 삭제도 타입 변경도 아니죠. 그런데 클라이언트 파싱 라이브러리에 따라 이게 **브레이킹 체인지**가 되는 것입니다. 게다가 그 앱은 고객사 앱이라 직접 테스트할 수 없었고, **배포 후에야 장애 리포트로** 알게 됐습니다.

## 버전을 올린다 — 그런데 버전도 만능은 아니다

이후로는 API 스펙이 변경되면 무조건 **버전을 올립니다.** 그런데 버전에도 비용이 있습니다.

- V1을 쓰는 클라이언트가 업데이트하지 않으면 **V1은 영원히 살아 있어야** 합니다.
- 버전이 쌓일수록 서버는 **동일 로직의 분기를 계속 유지**해야 합니다.
- 특히 **강제 업데이트가 불가능한 고객사 앱** 환경에서는 deprecated 버전을 끌고 가는 기간이 길어집니다.

## 진짜 방어 — 계약(Contract)

그래서 API 명세에 **"알 수 없는 필드는 무시해야 한다"를 명시**하는 게 중요합니다. 서버가 코드로 강제할 수 없는 영역이기 때문에, **계약으로 잡아 두는** 것입니다.

## 정리

> API 하위 호환성은 **서버 코드가 아니라 가장 엄격한 클라이언트 기준**으로 판단해야 합니다. 서버가 추가한 필드 하나가 누군가에겐 장애입니다.
