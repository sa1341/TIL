# TLS 하이브리드 암호화 (대칭키+비대칭키)

HTTPS로 데이터를 보낼 때 암호화 방식이 두 개 쓰입니다. **비대칭키 암호화**와 **대칭키 암호화**. 비대칭키가 더 안전하다고 배웠는데, 왜 그것만 안 쓰고 대칭키를 섞을까요?

![TLS 하이브리드 — 비대칭키로 세션키 교환, 대칭키로 데이터 암호화](../../../Attached%20file/net_tls_hybrid.svg)

## 두 방식의 차이

- **대칭키**는 빠르지만, 서버와 클라이언트가 **같은 키를 공유**해야 합니다. 처음 연결할 때 이 키를 네트워크로 보내야 하는데, 중간에 누가 가로채면 끝입니다.
- **비대칭키**는 이 **키 전달 문제를 해결**합니다. 공개 키는 누구나 가져가도 됩니다. 그걸로 암호화한 건 개인 키 없이는 못 푸니까요.

## 그런데 비대칭은 너무 느리다

비대칭 암호화는 대칭보다 **수백 배 이상 느립니다.** 웹페이지 하나 로딩할 때 수백 개의 리소스를 받는데, 전부 비대칭키로 암호화하면 페이지 로딩이 몇 분씩 걸립니다.

## TLS = 두 방식의 장점만 조합

- **비대칭키**는 처음 한 번, **세션 키를 교환할 때만** 쓰입니다.
- 이후 **실제 데이터는 전부 대칭키**로 암호화합니다.

즉 **느린 연산은 한 번만, 빠른 연산은 계속** 쓰는 것입니다.

## TLS 1.3에서는 이마저도 바뀌었다

- TLS 1.3은 **RSA 키 교환을 아예 제거**하고 **(EC)DHE(임시 키)만 허용**합니다.
- 매 연결마다 **임시 키쌍(ephemeral)**을 만들어서, 서버의 개인 키가 유출돼도 **과거 통신 내용은 복호화할 수 없게** 했습니다.
- 이걸 **순방향 비밀성(Forward Secrecy)**이라고 합니다.

## 정리

> HTTPS가 두 암호화를 섞는 건 **타협이 아니라, 각각의 약점을 상대방의 강점으로 메우는 설계**입니다. 비대칭키로 안전하게 키를 교환하고, 대칭키로 빠르게 데이터를 나릅니다.
