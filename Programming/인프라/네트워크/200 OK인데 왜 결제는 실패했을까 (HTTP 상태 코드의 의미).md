# 200 OK인데 왜 결제는 실패했을까 (HTTP 상태 코드의 의미)

결제 연동을 했는데, **주문은 실패인데 완료 알림이 나갔습니다.** HTTP 코드는 `200`이었죠. 서버는 요청을 정상적으로 받아 정상적으로 응답한 것이니까요. 그런데 **비즈니스 결과는 실패**였습니다. 상태 코드만 보고 성공 처리한 게 문제였습니다.

## 왜 이렇게 설계할까 — 200을 내리는 이유

이런 패턴이 생각보다 많습니다. 이유가 세 가지 있습니다.

1. **HTTP 상태 코드는 전송 계층의 결과다.** "요청은 잘 받았고 잘 처리했다"는 뜻이지, **비즈니스가 성공했다는 뜻이 아닙니다.**
2. **레거시 호환.** 오래된 시스템은 400번대 응답을 받으면 에러로 간주하고 재시도하거나 연결을 끊어 버립니다.
3. **프록시/CDN의 간섭.** 일부 프록시는 400번대 응답을 캐싱하거나, 자체 에러 페이지를 끼워 넣기도 합니다.

## 그래서 클라이언트가 방어해야 한다

외부 API를 연동할 때는 **상태 코드만 믿으면 안 됩니다.** 반드시 **응답 바디의 비즈니스 코드까지 확인**해야 합니다.

## "제대로 설계하자" — 그런데 여기도 함정

반대로 우리 API는 REST 원칙에 맞게 에러를 400·500번대로 내리는 방법도 있습니다. 잔액 부족은 `422`, 인증 실패는 `401`로 명확하게 내리는 것이죠.

그런데 이것도 함정이 있습니다. **상태 코드를 세분화할수록 인프라 장비가 간섭합니다.**

- 잔액 부족은 서버 장애가 아닌데, `5xx`를 많이 내리면 **서킷 브레이커가 서비스 자체를 끊어** 버릴 수 있습니다.
- 인프라 장비는 상태 코드 **숫자만 보지**, 그게 비즈니스 에러인지 시스템 에러인지 구분하지 않기 때문입니다.

그래서 실무에서는 **비즈니스 에러는 200 + 바디 코드, 시스템 에러는 4xx/5xx**로 나누는 팀이 많습니다.

## 정리

> **HTTP 상태는 시스템의 상태를 나타내는 것이지, 비즈니스의 성공 여부를 나타내는 게 아닙니다.** 외부 API 연동 시 상태 코드를 절대 맹신하지 마세요.
