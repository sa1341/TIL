# 롤링 업데이트 설정

롤링 업데이트는 **서비스 중단 없이** 애플리케이션을 새 버전으로 교체하는 매우 일반적인 배포 전략입니다. 구버전 Pod를 조금씩 줄이면서 신버전 Pod를 조금씩 늘려, 전체 용량을 유지한 채 점진적으로 갈아끼웁니다.

![롤링 업데이트 과정|697](../../../Attached%20file/k8s_rolling_update.svg)

롤링 업데이트 동작을 조정하는 매개변수는 **maxUnavailable**과 **maxSurge**입니다.

## maxUnavailable 설정

maxUnavailable 매개변수는 롤링업데이트 과정에서 사용 불가능한 파드의 최대 개수를 설정합니다.
절댓값 또는 백분율로 설정이 가능하며, 일반적으로 백분율을 사용하는 것이 디플로이먼트에서 원하는 복제본 수에 관계없이 사용 불가능한 파드의 수를 올바르게 적용할 수 있기 때문에 대부분의 서비스에 적합한 접근 방식입니다.

본질적으로 maxUnavailable 매개변수는 롤링 업데이트 진행 속도를 조정하는 데 도움을 줍니다.
만약 maxUnavailable 값을 50%로 설정하면 롤링업데이트는 기존 레플리카셋을 2개의 복제본으로 축소합니다.

그 후 롤링업데이트는 새로운 레플리카셋을 최대 2개의 복제본으로 확장해 축소로 제거된 파드를 대체해 총 4개의 복제본을 유지합니다.
그런 다음 기존 레플리카셋을 0개의 복제본으로 축소해 총 2개의 신규 복제본을 유지합니다. 
결국엔 50%로 설정하면 롤아웃이 4단계로 완료되며, 롤아웃 과정에서 50%의 가용률만 갖게 됩니다.


## maxSurge 설정

위에서 설명한 maxUnavailable과 마찬가지로 maxSurge는 특정 값이나 백분율로 지정 가능합니다.
maxSurge 매개변수는 롤아웃을 수행하고자 생성할 수 있는 추가 리소스의 양을 제어합니다. 
예를 들어서 10개의 복제본이 있는 서비스가 있다고 가정해보겠습니다.
maxUnavailable을 0으로 설정하고 maxSurge를 20%로 설정했습니다. 롤아웃 수행 시 첫 번째 단계는 새로운 레플리카셋에서 2개의 복제본을 추가로 생성해 서비스에서 총 12개의 복제본을 유지합니다.

그런 다음 이전 레플리카셋을 8개로 축소해 서비스에서 총 10개의 복제본을 유지합니다. 이러한 과정을 롤아웃이 완료될 때까지 진행합니다. 어느 시점에서나 서비스 용량은 최소 100%를 유지하며 최대 추가 리소스는 전체 리소스의 20%로 제한합니다.

> maxSurge를 100%로 설정하면 디플로이먼트 컨트롤러가 먼저 새로운 버전을 이전 버전의 100%까지 확장한 뒤, 새 버전이 안정화되면 이전 버전을 0%로 축소해 업데이트를 완료합니다. 신버전을 통째로 먼저 띄운다는 점에서 블루/그린과 닮았지만 **같지는 않습니다** — 진짜 블루/그린은 두 환경을 동시에 두고 **서비스 셀렉터로 트래픽을 한 번에 전환**하며 즉시 롤백할 수 있는 반면, 롤링(maxSurge 100%)은 Pod가 준비되는 대로 교체되어 잠깐 **두 버전이 함께 트래픽을 받는 구간**이 생깁니다.

## 두 가지 스타일 — 초과형 vs 감소형

같은 롤링 업데이트라도 두 값을 어떻게 잡느냐에 따라 "새 걸 먼저 띄우느냐" vs "헌 걸 먼저 내리느냐"가 갈립니다.

- **초과형 (maxSurge > 0 · maxUnavailable = 0):** 신버전을 **먼저 더 띄운 뒤** 구버전을 내립니다. 배포 중에도 가용 용량이 안 줄어 가장 안전하지만, 리소스를 더 씁니다(노드 여유 필요).
- **감소형 (maxSurge = 0 · maxUnavailable > 0):** 구버전을 **먼저 줄이고** 채웁니다. 리소스는 아끼지만 배포 중 용량이 잠깐 줍니다. (둘 다 0으로는 둘 수 없습니다.)

## 기본값과 반올림 함정 (replicas 2개일 때)

쿠버네티스 기본값은 `maxSurge 25%` + `maxUnavailable 25%`인데, `maxSurge`는 **올림**, `maxUnavailable`은 **내림**입니다.

- Pod가 2개면: `maxSurge = ceil(0.25 × 2) = 1`, `maxUnavailable = floor(0.25 × 2) = 0`.
- 즉 기본값이면 **최대 3개까지만** 뜨고 **한 개씩** 교체됩니다.
- 배포 중 **총 4개(구2 + 신2)가 동시에 뜨는 걸 봤다면 `maxSurge`를 2(또는 100%)로 명시적으로 올려둔 것**입니다.

```yaml
# 신버전 전체를 먼저 띄운 뒤 구버전을 내리는 설정 (용량 잠깐 2배, 무중단 최우선)
spec:
  replicas: 2
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 100%        # 또는 2
      maxUnavailable: 0
```

> replicas가 작으면 기본값 반올림 때문에 의도와 다르게 동작할 수 있으니 **값을 명시**하는 편이 안전합니다.
