# 쿠버네티스 오브젝트

쿠버네티스는 "무엇을 원하는가(desired state)"를 **오브젝트**로 선언하면, 컨트롤러가 현재 상태를 그 목표에 맞춰 유지해 줍니다. 핵심 오브젝트들은 아래처럼 계층적으로 연결됩니다.

![쿠버네티스 핵심 오브젝트 관계|697](../../../Attached%20file/k8s_objects.svg)

## 핵심 오브젝트

- **Pod**: 쿠버네티스의 **최소 배포 단위**로, 하나 이상의 컨테이너를 묶은 것입니다. Docker의 최소 단위는 컨테이너지만, k8s는 컨테이너 + 네트워크 + 스토리지를 묶은 **Pod** 단위로 배포합니다. Pod는 고유 IP를 갖지만 **휘발성**이라 언제든 사라지고 새로 만들어질 수 있습니다.
- **ReplicaSet**: 지정한 개수(`replicas`)만큼 Pod가 항상 떠 있도록 유지합니다. Pod가 죽으면 새로 만들어 개수를 맞춥니다.
- **Deployment**: ReplicaSet을 관리하며 **배포·롤링 업데이트·버전 롤백**을 담당합니다. 실무에서 Pod를 직접 만들기보다 대개 Deployment로 배포합니다.
- **Service**: 아래에서 설명.

> 관계 요약: **Deployment → ReplicaSet → Pod → 컨테이너**

## Service

Pod는 고유 IP를 가지지만, 다른 노드로 재배포되거나 휘발성으로 사라지면서 **IP가 계속 바뀝니다.** 그래서 변하지 않는 고정 진입점이 필요한데, 그것이 **Service**입니다. Service는 **라벨 셀렉터(예: `app=web`)**로 대상 Pod들을 찾아 트래픽을 분산 전달합니다.

Service 타입은 총 3가지입니다.

| 타입 | 용도 |
|---|---|
| **ClusterIP** (기본) | 클러스터 **내부**에서 Pod끼리 통신할 때 사용. 외부 노출용이 아님 |
| **NodePort** | 각 노드의 특정 포트를 열어 **외부에서** 접근 가능하게 함 |
| **LoadBalancer** | 클라우드 제공자의 로드 밸런서(예: AWS ELB)를 붙여 외부에 노출 |

즉 ClusterIP는 내부 통신, NodePort/LoadBalancer는 외부 노출을 담당한다고 이해하면 됩니다.
