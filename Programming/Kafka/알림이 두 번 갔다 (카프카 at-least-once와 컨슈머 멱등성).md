# 알림이 두 번 갔다 (카프카 at-least-once와 컨슈머 멱등성)

결제 한 건에 **알림톡이 두 번** 갔다는 CS가 들어왔습니다. 컨슈머는 분명 한 번만 처리하도록 짰는데요. 로그를 까 보니 **발송 직후 컨슈머가 죽었고**, 리밸런싱으로 다른 컨슈머가 **같은 메시지를 다시** 가져갔습니다.

![카프카 at-least-once — 처리와 커밋 사이 장애로 인한 중복](../../Attached%20file/kafka_at_least_once.svg)

## 카프카의 기본은 at-least-once

카프카의 기본 보장은 **at-least-once(최소 한 번)**입니다. 뒤집으면 **두 번 이상 올 수 있다**는 얘기입니다.

컨슈머 처리 흐름은 ① 메시지 수신 → ② 처리(알림 발송) → ③ 오프셋 커밋입니다. **문제는 ②와 ③ 사이**입니다.

- 알림은 이미 나갔는데 **오프셋을 커밋하기 직전에 컨슈머가 죽으면**, 브로커 입장에선 이 메시지가 **아직 처리 안 된 것**으로 남습니다.
- 리밸런싱으로 다른 컨슈머가 같은 메시지를 다시 집어가고, **알림은 한 번 더** 나갑니다.

## 해결 — 컨슈머를 멱등하게

해결책은 **컨슈머를 멱등(idempotent)하게** 만드는 것입니다. 같은 메시지가 두 번 와도 **결과가 한 번 처리한 것과 같도록** 만드는 것이죠.

## "Exactly-Once 모드 켜면 되잖아?" — 범위가 좁다

카프카의 **EOS(Exactly-Once Semantics)**는 사실 범위가 좁습니다.

- **카프카에서 읽어 카프카로 쓰는 파이프라인 안에서만** 보장됩니다. (**Kafka Streams**가 대표적)
- 그런데 우리 컨슈머가 하는 일은 보통 **외부 DB 쓰기, 외부 API 호출, 이메일 발송**입니다. 이건 카프카 트랜잭션 바깥이라 **EOS로 못 묶습니다.**
- DB 트랜잭션과 카프카 오프셋 커밋을 함께 묶고 싶다면 **아웃박스 패턴(Outbox)** 같은 추가 설계가 필요합니다.

## 정리

> Exactly-Once는 마법이 아니라 **프로듀서·브로커·컨슈머·저장소가 모두 협력해야 하는 프로토콜**입니다. 그래서 **컨슈머 멱등성은 어차피 필수 옵션**입니다.
>
> 메시지 큐를 쓴다는 건 "중복을 허용한다"가 아니라, **"중복을 처리할 책임을 컨슈머가 진다"는 약속**입니다.
