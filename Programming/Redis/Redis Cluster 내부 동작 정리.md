## 1. 슬롯 구조

Redis Cluster는 전체 키 공간을 **16384개 해시 슬롯**(0~16383)으로 나누고, 이걸 마스터들끼리 나눠 갖는다.

마스터 3대 기준 분배 예시:

```
master A: 슬롯 0     ~ 5460
master B: 슬롯 5461  ~ 10922
master C: 슬롯 10923 ~ 16383
```

**슬레이브는 슬롯을 독립적으로 소유하지 않는다.** 슬레이브는 특정 마스터 하나에 붙은 복제본(replica)이고, 그 마스터가 담당하는 슬롯 데이터의 **복사본**을 복제로 들고 있을 뿐이다. 슬롯 소유권은 오직 마스터에게 있다.

- slaveA는 masterA를, slaveB는 masterB를, slaveC는 masterC를 1:1로 복제한다.
- slaveA는 "0~5460 슬롯의 복제본"을 가질 뿐, 그 슬롯을 소유한 게 아니다.
- 기본적으로 슬레이브는 읽기 요청도 받지 않는다. 특정 슬롯의 읽기·쓰기는 그 슬롯을 **소유한 마스터**로 간다. 복제본에서 읽으려면 클라이언트가 `READONLY`를 걸어야 한다.

> 최신 Redis는 "slave" 대신 "replica" 용어를 쓰고 설정도 `cluster-replica-*`로 바뀌었다(옛 `cluster-slave-*`는 alias로 동작).

## 2. Failover 시 마스터 승격

승격은 **"누가 제일 최신 데이터를 가졌나 + 마스터들의 과반 투표"**로 결정된다.

### 흐름

```
마스터 다운 (node-timeout 내 무응답)
        │
        ▼
PFAIL → FAIL (과반 마스터가 합의)
        │
        ▼
복제본 선거 대기 (지연 = 복제 offset 순위)
        │
        ▼
투표: 마스터 과반 필요 (epoch당 1표)
        │
        ▼
승격 → 슬롯 인수 (새 configEpoch 전파)
```

### 단계별 설명

**① 장애 감지.** 노드들은 cluster bus로 PING/PONG 하트비트를 주고받는다. 어떤 마스터가 `cluster-node-timeout` 안에 응답을 안 하면 그걸 발견한 노드가 **PFAIL**(주관적 의심)로 표시한다. 이 정보가 gossip으로 퍼져 **과반의 마스터**가 같은 노드를 PFAIL로 보고하면 **FAIL**(객관적 확정)로 승격된다. 한 노드의 착각으로는 failover가 일어나지 않는다.

**② 선거 대기 (승격자를 가르는 핵심).** 슬레이브들은 마스터가 FAIL이 돼도 바로 선거에 뛰어들지 않고 짧은 지연을 기다린다.

```
지연 = 500ms 고정 + 랜덤(0~500ms) + 순위(rank) × 1000ms
```

여기서 순위는 **복제 오프셋(replication offset)** 기준이다. 죽은 마스터의 데이터를 가장 많이 받아둔 복제본이 순위 0(가장 짧은 대기)이 되어 먼저 선거를 시작한다. 결과적으로 **데이터가 가장 최신인 복제본이 마스터가 될 확률이 높아진다.** 유실 최소화 설계.

**③ 투표.** 대기가 끝난 복제본은 currentEpoch를 올리고 모든 마스터에게 `FAILOVER_AUTH_REQUEST`를 브로드캐스트한다. 각 마스터는 한 epoch당 딱 한 표만 주고(`FAILOVER_AUTH_ACK`), FAIL 상태인 마스터의 복제본에게만 준다. 이 복제본이 **마스터 과반의 표(N/2 + 1)**를 모으면 선거에서 이긴다.

**④ 승격.** 승자는 스스로를 마스터로 승격하고, 죽은 마스터가 갖던 슬롯들을 자기 것으로 선언한 뒤, 더 높은 configEpoch를 실어 PONG을 뿌려 나머지 노드들이 슬롯 맵을 갱신하게 만든다.

### 실무상 반드시 알아야 할 조건

- **마스터 과반이 살아서 서로 통신 가능해야** failover가 된다. 네트워크 파티션으로 마스터 과반을 확보하지 못한 쪽은 투표를 못 모아 승격이 안 되고 쓰기를 중단한다(split-brain 방지). → 마스터 수를 **홀수**로 두고 과반(quorum)을 확보하는 게 중요.
- **`cluster-replica-validity-factor`** (기본 10): 복제본이 마스터와 너무 오래 끊겨 있었으면 데이터가 심하게 뒤처졌다고 보고 자동 승격 대상에서 제외한다. 판정식은 `끊긴 시간 > cluster-node-timeout × validity-factor`. 예: node-timeout 15초 × 10 = 150초 이상 끊긴 복제본은 선거를 시작하지 않는다. `0`으로 두면 아무리 뒤처져도 승격을 시도한다(가용성↑, 일관성↓ 트레이드오프).
- **`replica-priority`**: 값이 `0`인 복제본은 절대 마스터로 승격되지 않는다(예: 백업/분석 전용 복제본, DR 사이트 노드를 승격 대상에서 빼고 싶을 때). 값이 낮을수록 우선.

## 3. 죽은 마스터의 슬롯 데이터 이관

**결론: failover 시점에 데이터 이관(복사·이동)은 일어나지 않는다.**

승격된 복제본은 평소에 마스터를 계속 복제하고 있었기 때문에 **그 데이터를 이미 갖고 있다.** 바뀌는 건 데이터가 아니라 **슬롯 소유권**이다. 새 마스터가 configEpoch를 올려 "이 슬롯들은 내 거"라고 선언하고, 그 슬롯→노드 매핑이 gossip으로 클러스터 전체에 갱신될 뿐이다. 데이터는 원래 복제본에 있었고, 라벨만 바뀌는 셈.

### 함정: 비동기 복제로 인한 유실

Redis 복제는 **비동기**다. 죽은 마스터가 클라이언트에게 "OK"를 보냈지만 아직 복제본으로 넘기지 못한 쓰기가 있었다면, failover 시 그 쓰기는 **유실된다.** Redis Cluster는 강한 일관성을 보장하지 않는다. `WAIT` 명령으로 복제 확인을 강제해 이 창을 줄일 수는 있지만 완전히 없앨 수는 없다.

### 경계 케이스

- **슬레이브 없는 마스터가 죽으면?** 그 슬롯들은 인수할 복제본이 없어 커버리지가 사라진다. 기본값 `cluster-require-full-coverage yes`면 슬롯 하나라도 비는 순간 클러스터가 down 상태(`cluster_state:fail`)가 되어 `CLUSTERDOWN` 에러를 반환한다(공식 문서 표현으로는 "쓰기를 멈춘다"이고, down 상태에선 읽기도 기본 차단된다). `no`로 두면 살아있는 슬롯은 계속 서비스하고 죽은 슬롯만 에러. 참고로 `cluster-allow-reads-when-down yes`를 주면 down 상태여도 커버되는 슬롯의 읽기는 허용할 수 있다. → 마스터마다 슬레이브를 붙이는 이유가 이 자동 failover 능력 확보.
- **죽었던 마스터가 부활하면?** 자기가 이미 대체당한 걸(다른 노드가 더 높은 configEpoch를 가진 걸) 발견하고 스스로 **새 마스터의 슬레이브로 강등**된다. 이후 재동기화를 받는다.

## 4. cluster bus

노드끼리 대화하는 **별도 통신 채널**. 각 노드는 포트를 두 개 연다.

| 포트 | 용도 |
| --- | --- |
| 클라이언트 포트 (예: 6379) | 클라이언트가 `GET`/`SET`을 RESP 프로토콜로 보내는 곳 |
| 클러스터 버스 포트 (기본 = 클라이언트 포트 + 10000, 예: 16379) | **노드끼리만** 쓰는 채널 |

- 버스 포트는 기본이 "클라이언트 포트 + 10000"이지만 `cluster-port` 설정으로 임의 지정할 수 있다(컨테이너/포트 매핑 환경에서 유용).
- 버스는 RESP가 아니라 **바이너리 프로토콜**을 쓴다. 하트비트가 상시 오가기 때문에 가볍게 설계됐다.
- 이 채널로 **gossip**이 오간다. 각 노드는 매초 몇몇 노드에게 PING을 보내며, 그 패킷 안에 자기가 아는 다른 노드들의 상태(PFAIL 의심 여부, configEpoch, 소유 슬롯 등)를 **얹어서(piggyback)** 같이 보낸다. 이렇게 정보가 번져 중앙 코디네이터 없이 서로의 상태를 알게 된다.
- 하트비트 외 특수 메시지도 이 버스로: **FAIL**(장애 확정), **FAILOVER_AUTH_REQUEST/ACK**(투표), **MEET**(노드 추가), **UPDATE**(오래된 설정 수정) 등.

> **운영 주의:** 버스 포트는 클라이언트에 노출하면 안 된다. 방화벽으로 **클러스터 노드끼리만** 접근 가능하게 막을 것.

## 5. configEpoch

각 마스터가 들고 있는 **설정 버전 번호**. 정확히는 "이 슬롯들은 내 거"라는 소유권 주장에 붙는 버전.

**존재 이유는 하나 — 슬롯 소유권 충돌 해결.** failover·파티션·재시작이 겹치면 두 노드가 같은 슬롯을 서로 자기 거라고 주장할 수 있다. 이때 클러스터는 이력을 따지지 않고 **configEpoch가 더 높은 쪽을 소유자로 인정한다.** 그냥 숫자 큰 쪽이 이긴다.

> 비유: 낙관적 락의 `@Version` 컬럼과 같은 원리. 누가 최신인지 판단할 때 전체 히스토리 재구성 없이 버전 숫자만 비교한다. (분산 시스템의 고전적 방법 — Lamport clock)

### "올린다"는 게 무슨 의미인가

1. masterA가 configEpoch **5**로 슬롯 0~5460을 소유하다 죽는다.
2. slaveA가 선거에서 이기면 5보다 높은 **새 configEpoch 6**을 받아 "이제 0~5460은 내 거"라고 선언한다. 이 6이 주장을 **권위 있게** 만드는 장치.
3. 이 주장이 버스로 퍼지면 나머지 노드들이 "6 > 5"이므로 슬롯 맵을 새 마스터로 갱신한다.
4. 나중에 masterA가 부활해 자기 configEpoch 5로 우겨도, 이미 6을 가진 새 마스터가 있으니 `5 < 6`이라 짐을 인정하고 슬레이브로 강등된다.

즉 failover 마지막 단계 "새 configEpoch 전파"가 하는 일 = 데이터를 옮기는 게 아니라, **더 높은 버전 번호로 슬롯 소유권의 최신성을 선언**해 클러스터 전체가 새 마스터를 인정하게 만드는 것.

> configEpoch 유니크성: 선거 때 쓰는 currentEpoch가 단조증가하고 승격에 마스터 과반 동의가 필요해 보통 마스터마다 겹치지 않는다. 겹치는 상황이 생기면 node ID가 작은 쪽이 자기 configEpoch를 올려 자동 해소한다.

## 6. 슬롯 맵과 라우팅

슬롯 맵은 **"어느 슬롯을 어느 노드가 소유하는가"를 적어둔 표**. 16384개 슬롯 각각이 어떤 마스터 것인지의 매핑.

```
슬롯 0     → masterA
슬롯 1     → masterA
...
슬롯 5460  → masterA
슬롯 5461  → masterB
...
슬롯 16383 → masterC
```

**핵심: 이 표를 클러스터의 모든 노드가 각자 자기 메모리에 사본으로 들고 있다.** 중앙 서버에 하나만 있는 게 아니다. gossip이 바로 이 표를 서로 맞추는 과정.

### 클라이언트 요청 라우팅

클라이언트가 `SET user:1000 ...`을 하면:

1. 키를 `CRC16(key) % 16384`로 해싱해 슬롯 번호를 구함 (예: 슬롯 5000)
2. 슬롯 맵을 봐서 "슬롯 5000은 masterA 소유" 확인
3. masterA로 요청 전송

엉뚱한 노드로 보내면 그 노드는 `MOVED 5000 <masterA-주소>`를 돌려준다. 스마트 클라이언트는 이걸 받으면 슬롯 맵을 갱신하고 올바른 노드로 다시 보낸다. → **슬롯 맵 = "이 키는 누구한테 물어봐야 하나"의 라우팅 테이블.**

### "슬롯 맵 갱신"의 의미

failover로 masterA가 죽고 slaveA가 승격되면 슬롯 0~5460의 소유자가 바뀐다. 모든 노드가 각자 들고 있던 슬롯 맵의 해당 부분을 고쳐 써야 한다:

```
갱신 전:  슬롯 0~5460 → masterA
갱신 후:  슬롯 0~5460 → slaveA(새 마스터)
```

즉 **각 노드가 자기 메모리의 슬롯→노드 매핑 표에서 죽은 마스터의 슬롯 소유자를 새 마스터로 바꿔 적는 것.** configEpoch 비교로 더 높은 버전이 이기며, 이 갱신이 gossip으로 수백 ms 내 전체로 번지면 클라이언트도 MOVED를 받아 새 마스터로 향한다.

**전체 연쇄:** 소유권 변경(승격) → 더 높은 configEpoch로 선언 → gossip 전파 → 각 노드가 슬롯 맵 수정 → 클라이언트 라우팅이 새 마스터를 향함

## 7. 클라이언트(Lettuce / Redisson)와 슬롯 맵

**Lettuce·Redisson은 슬롯 맵을 클라이언트 메모리에 캐싱하는 스마트 클라이언트다.**

| 방식 | 동작 |
| --- | --- |
| 덤(dumb) 클라이언트 | 슬롯 맵 없음. 아무 노드에나 보내고 MOVED가 오면 리다이렉트 → 매번 2회 왕복 가능 |
| 스마트 클라이언트 (Lettuce, Redisson, JedisCluster) | 슬롯 맵을 미리 받아 캐싱. 키 해싱해 **처음부터 올바른 마스터로 직접** 전송 |

클라이언트가 표를 들고 있으면 MOVED 왕복 없이 한 번에 담당 마스터로 쏠 수 있다.

### 표를 얻고 갱신하는 방법

- **최초 획득:** 부팅 시 시드 노드에 `CLUSTER SHARDS`를 날려 전체 토폴로지를 받아 슬롯 맵을 구성한다. (`CLUSTER SLOTS`도 같은 용도지만 공식적으로 **deprecated**라 신규 코드는 `CLUSTER SHARDS`를 쓰는 게 맞다. 다만 다수 클라이언트가 하위호환으로 여전히 SLOTS를 호출한다.)
- **갱신 트리거** (클라이언트는 cluster bus에 안 껴서 gossip을 못 받음):
  - **MOVED 응답을 받았을 때** — 표가 낡았다는 신호로 받아 슬롯 맵을 다시 받아온다(토폴로지 리프레시).
  - **주기적 리프레시** — 백그라운드로 일정 주기마다 토폴로지를 재조회.

### Lettuce vs Redisson (실무 포인트)

**Lettuce** — 토폴로지 자동 리프레시가 기본 **꺼져 있다.** 명시적으로 켜지 않으면 failover 후에도 클라이언트 슬롯 맵이 안 바뀌어 죽은 노드로 계속 보내다 에러가 날 수 있다.

```java
ClusterClientOptions.builder()
    .topologyRefreshOptions(
        ClusterTopologyRefreshOptions.builder()
            .enablePeriodicRefresh(Duration.ofSeconds(30))   // 주기적 리프레시
            .enableAllAdaptiveRefreshTriggers()              // MOVED/ASK 등 발생 시 리프레시
            .build())
    .build();
```

> Spring Boot에서는 `LettuceClientConfigurationBuilderCustomizer`로 이 옵션을 주입하는 패턴을 사용. 운영 환경이면 이게 켜졌는지 반드시 확인.

**Redisson** — 토폴로지 스캔이 기본 내장. `scanInterval`(기본 1000ms) 주기로 노드 상태를 확인하고 알아서 갱신한다. 이 부분 기본값이 Lettuce보다 안전한 편.

### MOVED vs ASK

| 리다이렉트 | 의미 | 클라이언트 동작 |
| --- | --- | --- |
| **MOVED** | 슬롯 소유자가 **영구적으로** 바뀜 (failover, 리샤딩 완료) | 슬롯 맵을 **갱신** |
| **ASK** | 리샤딩 **진행 중**이라 그 키만 일시적으로 다른 노드에 있음 | **이번 요청만** 그쪽으로 보냄. 슬롯 맵은 **갱신 안 함** |

MOVED = "표를 고쳐라", ASK = "이번만 저기 가봐라".

## 8. 리샤딩과 토폴로지

### 토폴로지(topology)

**클러스터의 전체 구성도.** "어떤 노드가 몇 대 있고, 각각 마스터냐 복제본이냐, 누가 누구의 복제본이고, 어떤 슬롯을 소유하는가" 전체.

```
masterA (슬롯 0~5460)      ← slaveA가 복제
masterB (슬롯 5461~10922)  ← slaveB가 복제
masterC (슬롯 10923~16383) ← slaveC가 복제
```

슬롯 맵은 이 토폴로지의 **일부**(슬롯→노드 매핑만). 토폴로지는 거기에 더해 노드 역할, 복제 관계, 노드 주소까지 포함. `CLUSTER SHARDS`(현행) 또는 `CLUSTER SLOTS`(deprecated), 그리고 `CLUSTER NODES`가 이 토폴로지를 응답으로 주는 명령. Lettuce의 "토폴로지 리프레시" = 이 구성도를 최신으로 다시 받아오는 동작.

### 리샤딩(resharding)

**슬롯을 노드 사이에서 옮기는 작업.** "슬롯 몇 번~몇 번을 A 마스터에서 B 마스터로 이동".

**언제:**
- **노드 추가** — 새 마스터는 슬롯이 0개라 논다. 기존 마스터들에서 슬롯 일부를 떼어 옮겨야 부하를 나눈다.
- **노드 제거** — 뺄 마스터의 슬롯을 다른 마스터들로 전부 옮긴 뒤 제거.
- **부하 불균형 해소** — 핫한 슬롯이 몰린 마스터에서 수동 재분배.

**`redis-cli --cluster reshard` 내부 동작:**

1. 대상 슬롯을 소스는 `MIGRATING`, 목적지는 `IMPORTING` 상태로 표시
2. 슬롯에 속한 키들을 `MIGRATE` 명령으로 **하나씩 실제로 목적지에 복사·이동** (진짜 데이터가 네트워크로 넘어감)
3. 이동 중 아직 안 옮겨진 키 요청이 목적지로 오면 **ASK 리다이렉트**로 처리 (리샤딩 중엔 MOVED가 아니라 ASK)
4. 다 옮기면 `CLUSTER SETSLOT`으로 소유권을 확정하고, 이때부터 **MOVED**로 전환

### failover vs 리샤딩 (핵심 구분)

| | failover | 리샤딩 |
| --- | --- | --- |
| 계기 | 마스터 장애 (계획 밖 사고 대응) | 노드 추가/제거/균형 조정 (계획된 운영) |
| 슬롯 이동 | 소유권 라벨만 승계 | 슬롯을 의도적으로 이동 |
| 데이터 | **이동 없음** (복제본이 이미 보유) | `MIGRATE`로 **키를 실제로 옮김** |
| 리다이렉트 | MOVED | 진행 중 ASK → 완료 후 MOVED |

## 9. 슬롯 16384개 고정

슬롯 개수는 **항상 16384개로 고정.** 노드가 3대든 100대든 슬롯 총량은 언제나 16384. 바뀌는 건 개수가 아니라 **분배 방식.**

| 마스터 수 | 마스터당 슬롯 |
| --- | --- |
| 3 | 약 5461 |
| 4 | 약 4096 |
| 16 | 약 1024 |

노드를 추가하면 리샤딩으로 기존 슬롯을 재분배하는 것이지 슬롯을 새로 만드는 게 아니다. 총합은 항상 16384.

### 왜 하필 16384인가 (antirez의 이유)

**① 하트비트 메시지 크기.** 노드가 cluster bus로 주고받는 PING/PONG 안에 "내가 어떤 슬롯을 소유하는지"를 비트맵으로 싣는다. 이게 매 하트비트마다 오간다.

- 16384 슬롯 → 16384 비트 = **2KB**
- 65536(2^16) 슬롯이었다면 → 8KB

상시 오가는 하트비트에 8KB는 너무 무겁다. 2KB가 대역폭상 합리적인 선.

**② 노드 수는 어차피 그렇게 많지 않다.** Redis Cluster는 현실적으로 최대 1000노드 규모를 상정. 슬롯이 노드보다 충분히 많아야 부하를 고르게 나누는데, 1000노드 기준 16384 슬롯이면 노드당 16개씩이라 충분. 굳이 65536까지 늘려 하트비트만 무겁게 할 이유가 없었다.

→ 16384 = **하트비트 대역폭(2KB)과 부하 분산 세분화 사이의 절충점.**

## 10. 대용량 서비스의 노드 개수 설계

**노드 개수는 트래픽 숫자가 아니라 데이터 크기·처리량을 계산해 역산한다.** "대용량이면 몇 대"라는 고정 답은 없다.

전제: 마스터:슬레이브 1:1(총 노드 = 마스터 × 2)이 표준. 가용성이 더 중요하면 마스터당 복제본 2개(× 3).

### 실무 규모 감각 (절대 기준 아님)

| 규모 | 마스터 수 | 총 노드(1:1) |
| --- | --- | --- |
| 소규모 | 3 | 6 (최소 권장) |
| 중간 | 3~6 | 6~12 |
| 대용량 | 6~10 | 12~20 |
| 초대형 | 수십 | 수십~ |

**대용량이라고 마스터가 수십 대 가는 경우는 생각보다 드물다.** 마스터 한 대가 초당 수만~십수만 op를 소화하므로 트래픽만 보면 6~10 마스터로 웬만한 대형 서비스가 커버된다. 노드가 폭증하는 건 보통 트래픽이 아니라 **데이터 양** 때문.

### 두 가지 병목 계산 (더 빡센 쪽이 결정)

**① 메모리(데이터 크기) 기준 — 대개 실제 병목**

```
필요 마스터 수 = 전체 데이터셋 크기 / 마스터 1대의 실효 메모리
```

실효 메모리는 인스턴스 메모리를 그대로 쓰면 안 되고:
- 마스터는 가용 메모리의 **60~70%까지만** 데이터로 채운다(`maxmemory`를 그 선에 설정). 나머지는 복제 버퍼, 리샤딩 임시, fork 시 copy-on-write 오버헤드용.
- 예: 64GB 인스턴스 → 실효 데이터 40GB로 잡는 게 안전.

예시: 데이터셋 500GB, 인스턴스 64GB(실효 40GB) → 마스터 약 13대 → 총 26노드.

**② 처리량(throughput) 기준**

```
필요 마스터 수 = 피크 QPS / 마스터 1대가 감당하는 QPS
```

- 기본적으로 읽기·쓰기 모두 마스터로 간다. 슬레이브는 복제만 하며 논다.
- `READONLY`를 걸면 복제본에서도 읽을 수 있어, **읽기가 압도적으로 많으면** 슬레이브를 read replica로 활용해 마스터 부담을 던다.
- 단 복제가 비동기라 복제본 읽기는 **stale read** 위험. 강한 일관성이 필요한 읽기는 마스터로.

### 노드를 늘리기 전에 점검할 것

- **핫키/핫슬롯 편중** — 특정 키에 트래픽이 몰리면 노드를 늘려도 그 키를 소유한 마스터 한 대만 터진다. 노드 수 문제가 아니라 키 설계 문제(샤딩 키 분할, 로컬 캐시).
- **hash tag 사용** — `{user:1000}:profile`처럼 중괄호를 쓰면 그 부분만 해싱돼 같은 슬롯에 묶인다. multi-key 연산(MGET, 트랜잭션, Lua)에 필요하지만 남발하면 슬롯 분포가 쏠려 특정 마스터에 데이터가 몰린다.
- **노드 증가의 대가** — 마스터가 늘면 gossip 트래픽, 하트비트, failover 조율 비용이 커진다. 수백 노드는 운영 난도가 급상승. "노드 추가"보다 "더 큰 인스턴스로 scale-up"이 먼저 검토되기도 한다.

### 결론

대용량이라도 마스터 **6~10대(총 12~20노드)** 선에서 시작해 계산으로 조정하는 게 가장 흔하다. 그 숫자는 트래픽이 아니라 대개 **데이터셋 크기 ÷ 마스터당 실효 메모리(인스턴스의 60~70%)**로 역산해 나온다. 처리량이 병목이면 노드를 늘리기 전에 슬레이브 read 분산과 핫키 여부를 먼저 점검한다.

## 핵심 요약 치트시트

- **슬롯**: 항상 16384개 고정. 마스터가 소유, 슬레이브는 복제본만 보유(소유 아님).
- **Failover**: 과반 마스터가 FAIL 합의 → 복제 오프셋 최신 복제본이 우선 대기 → 마스터 과반 투표로 승격.
- **데이터 이관**: failover 때 데이터 이동 **없음**(복제본이 이미 보유), 소유권만 승계. 비동기 복제라 미복제 쓰기는 **유실** 가능.
- **cluster bus**: 클라이언트 포트 + 10000. 노드 전용 바이너리 채널. gossip이 여기서 오감. 클라이언트에 노출 금지.
- **configEpoch**: 슬롯 소유권 충돌을 푸는 버전 번호(`@Version`과 동일 원리). 높은 쪽이 소유자. 승격 시 올려서 새 소유권을 권위 있게 선언.
- **슬롯 맵**: 모든 노드가 각자 사본 보유하는 슬롯→노드 라우팅 표. 갱신 = 소유자를 새 마스터로 고쳐 적는 것.
- **클라이언트**: Lettuce/Redisson은 슬롯 맵 캐싱(스마트 클라이언트). **Lettuce는 토폴로지 리프레시 기본 OFF** → 운영 시 반드시 켤 것. MOVED=표 갱신, ASK=이번만.
- **리샤딩 vs failover**: 리샤딩은 계획된 슬롯 이동 + `MIGRATE`로 실제 데이터 이동. failover는 사고 대응 + 데이터 이동 없음.
- **노드 설계**: 데이터셋 크기 ÷ 마스터 실효 메모리(인스턴스 60~70%)로 역산. 대용량도 흔히 6~10 마스터. 핫키·hash tag 편중 먼저 점검.
