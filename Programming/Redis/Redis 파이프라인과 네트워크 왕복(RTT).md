# Redis 파이프라인과 네트워크 왕복(RTT)

상품 목록 API가 있습니다. 상품마다 Redis에서 좋아요 수를 가져옵니다. 상품이 10개일 때는 빨랐습니다. 그런데 **1,000개를 조회하니 API 응답이 3초**가 걸렸습니다. Redis는 빠르다면서요?

![Redis 파이프라인 — 1000번 왕복 vs 1번 배치|697](../../Attached%20file/redis_pipeline.svg)

## 병목은 Redis가 아니라 네트워크 왕복

- Redis 명령 자체는 **마이크로초 단위**로 처리됩니다.
- 하지만 명령 하나를 보내고 응답을 받는 데는 **네트워크 왕복 시간(RTT)**이 붙습니다. 같은 리전 안에서도 왕복이 **1~3ms**는 걸립니다.
- **1,000번 반복하면 네트워크 대기만 수 초**입니다. Redis가 아무리 빨라도 네트워크를 1,000번 왕복하면 느릴 수밖에 없습니다.

## 해결 — 파이프라인(Pipeline)

명령을 **하나씩 보내고 기다리는 게 아니라, 한꺼번에 모아서 보내고 응답도 한꺼번에** 받습니다.

- 네트워크 왕복이 **1,000번 → 1번**으로 줄어듭니다.
- 3초 걸리던 API가 **50ms 이내**로 줄어듭니다. 코드 몇 줄 바꿨을 뿐인데 수십 배 차이가 납니다.

## 파이프라인의 주의점

1. **메모리 압박** — 명령을 모아 한 번에 보내니 **응답도 한꺼번에 메모리에 쌓입니다.** 값이 큰 JSON이나 직렬화 객체라면 GC 압박이 심해집니다. → 대량 처리 시 **500~1,000개 단위로 끊어서** 실행하는 게 안전합니다.
2. **원자성이 없다** — 파이프라인 안의 명령들은 트랜잭션이 아닙니다. 중간에 하나가 실패해도 나머지는 그대로 실행됩니다. → 원자성이 필요하면 **`MULTI`/`EXEC`**로 묶고, 조건 분기까지 필요하면 **Lua 스크립트**를 씁니다.

## 정리

> Redis가 느리다면 Redis가 아니라 **네트워크를 의심**해야 합니다. 명령 수만큼 왕복하고 있다면, **파이프라인 하나로 네트워크 병목은 사라집니다.**
