# Redis 분산락 (동시성 · TTL · Watchdog)

서버를 두 대로 늘렸더니 **쿠폰이 초과 발급**됩니다. 코드엔 `synchronized`가 붙어 있는데도요. 이 노트는 왜 `synchronized`로는 부족하고, Redis 분산락이 어떻게 발전하며 어떤 한계를 갖는지 정리합니다.

## 1. 왜 synchronized로는 안 되나

- `synchronized`는 **같은 JVM 안에서만** 잠금입니다.
- 서버가 두 대면 **JVM도 두 개**입니다. 각자 자기 메모리를 독립적으로 잠그니, 서로의 존재를 모릅니다.
- 여러 스레드/프로세스가 동시에 같은 자원을 읽고 쓰면서 결과가 실행 순서에 따라 달라지는 문제 — **레이스 컨디션(Race Condition)**입니다.

해결하려면 **모든 서버가 공통으로 바라보는 외부 저장소**에 잠금을 걸어야 합니다. 이게 **분산락**이고, **Redis**가 이 역할을 가장 많이 맡습니다.

## 2. 분산락의 기본 원리

- Redis에 **"이 키가 없으면 내가 선점한다"**는 명령(`SET key value NX`)을 보냅니다.
- 여러 서버가 동시에 보내도, Redis는 **단일 스레드**라서 **하나만 성공**합니다. 나머지는 "이미 있음" 응답을 받고 대기하거나 포기합니다.

## 3. TTL — 생명줄이자 사고 원인

`TTL`(만료 시간)이 없으면, 락을 잡은 **서버가 갑자기 죽었을 때 락을 해제할 코드가 실행되지 않아** 그 락이 영원히 남습니다. 다른 서버는 영원히 대기 — **데드락**입니다. TTL이 있으면 서버가 죽어도 몇 초 후 락이 자동으로 풀립니다.

그런데 **바로 이 TTL 때문에 문제가 생깁니다.**

- 서버 A가 락을 잡고 작업하는데, 작업이 TTL(예: 3초)보다 오래 걸리면? 자바에서 **Full GC**가 발생하면 애플리케이션이 수 초간 멈춥니다. 평소 50ms면 끝나는 작업이 GC 때문에 4초가 걸릴 수 있죠. 그 사이 **TTL이 만료**됩니다.
- **짧게 잡으면** 정상 작업 중에 락이 풀려 버리고, **길게 잡으면** 서버가 죽었을 때 복구가 느려집니다.

## 4. 두 가지 해결책 — UUID 이름표 + Watchdog

**문제 A: TTL 만료 틈에 "남의 락"을 지운다.** 서버 A의 TTL이 만료된 뒤 서버 B가 새 락을 잡았는데, 서버 A가 작업을 끝내고 `finally`에서 락을 삭제하면 — 그건 **서버 B의 락**입니다.

- **해결: 락에 이름표(UUID)를 붙인다.** 각 서버가 고유한 UUID를 락에 저장하고, 해제할 때 **"내 이름표가 맞으면"** 삭제합니다.

**문제 B: 작업이 TTL보다 오래 걸린다.**

- **해결: Watchdog.** 작업이 진행 중이면 **TTL을 자동으로 연장**하는 별도 스레드를 둡니다. 서버가 살아 있으면 계속 연장하고, 죽으면 연장이 멈춰 TTL이 자연스럽게 만료됩니다. **Redisson** 라이브러리가 이 방식을 씁니다.

## 5. 그래도 남는 한계

- UUID로 남의 락 삭제를 막고 Watchdog으로 TTL 만료를 막으면 꽤 안전해 보이지만, **Redis 서버 자체가 죽으면 락 정보가 통째로 사라집니다.**
- 이를 보완하는 **Redlock** 알고리즘도 있지만, **네트워크 지연 앞에서는 완벽하지 않다는 반론**도 있습니다.

## 정리

> 분산락은 **동시 접근 확률을 낮추는 도구이지, 절대적인 보장이 아닙니다.** 정말 중요한 데이터라면 **DB의 유니크 제약 조건** 같은 최종 방어선을 반드시 함께 두어야 합니다.
